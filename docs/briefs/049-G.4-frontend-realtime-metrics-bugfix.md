## Fix A — realtime negative playback latency (HIGH, `/tdd`)
**Evidence:** MetricsPanel showed `speech-end → playback = -2102 ms` for a realtime turn — negative within ONE browser clock, so a **stamping bug**, not cross-clock disclosure.

**Diagnosis (loci):**
- `playback.started` stamped on the `<audio>` `onplaying` event — `web/src/realtime/realtimeWebRtcClient.ts:199`.
- `turn.recording.stopped` stamped on the Stop click — `web/src/realtime/realtimeTurnController.ts:178` (`marker('turn.recording.stopped')` → `clock()` L63).
- `turn.recording.started` — `realtimeTurnController.ts:163`.
- A negative `recording.stopped → playback.started` means `playback.started` was stamped **before** `recording.stopped`. Likely causes to pin: the `<audio>` `onplaying` firing for a **prior turn's** audio (reused element — lesson §17 detached `<audio>`), a **per-turn latch not reset**, or `playback.started` being stamped from a stale/earlier event. (In a correct manual VAD-off turn, audio comes back only AFTER Stop→commit→response.create, so playback.started MUST be ≥ recording.stopped.)

**Acceptance (A):**
- [ ] A **failing test** pins the correct stamping invariant for a realtime turn: given a normal event order (start → stop → first-audio), `playback.started ≥ recording.stopped` (the speech-end→playback delta is non-negative), and the playback stamp is **per-turn** (not leaked from a prior turn). RED first.
- [ ] Fix so the realtime first-audio stamp is captured against the **current** turn (reset/scope the latch per turn; stamp on the right event). If a genuinely-ambiguous async boundary remains, **disclose honestly** (the §13 cross-clock pattern) rather than emit a negative number — but the primary fix is correct per-turn stamping.
- [ ] `deriveTurnMetrics` math unchanged (it computes deltas from the stamps; the bug is the stamps, not the math).

## Fix B — MetricsPanel "Cascade stages" header shown for realtime (HIGH, `/tdd`)
**Evidence:** realtime.* events rendered beneath a hardcoded **"Cascade stages"** header.

**Diagnosis (loci):** `web/src/components/MetricsPanel.tsx` — the per-stage **bar** (~L113–138) is correctly gated `{mode === 'cascade' && …}`, but the **`turn-stages` section** (~L151) renders **unconditionally** with a hardcoded `"Cascade stages"` header (~L153). (Pre-existing logic latent until the first live realtime run; H.1b was CSS-only — confirm the gate was already absent, not removed by H.1b.)

**Acceptance (B):**
- [ ] A **failing test** pins: for a **realtime** turn, the panel does NOT render the "Cascade stages" / per-stage section (or renders a realtime-appropriate label, not "Cascade stages"). RED first.
- [ ] Fix: mode-gate the whole `turn-stages` section to `mode === 'cascade'` (preferred — realtime has no STT/Translation/TTS stages), OR conditionalize the header text per mode. **Preserve the `turn-stages` aria-label** for the cascade case (tests query it). Default: gate the section to cascade; for realtime show nothing there (the realtime headline speech→first-audio already renders above).
- [ ] Cascade turns still render the per-stage section unchanged (no regression).
