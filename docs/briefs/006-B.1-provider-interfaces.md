## RED test outline (Step 2 — `ProviderErrorMappingTests.cs`)
1. **`rate_limit_429_maps_retryable`** — (stage `stt`) → `Code=="stt.rate_limited"`, `Retryable==true`, `HttpStatusCode==429`. Why: ARCH-012 table.
2. **`auth_401_403_not_retryable`** — → `<stage>.auth`, `Retryable==false`, status preserved. Why: ARCH-012.
3. **`invalid_request_400_422_not_retryable`** — → `<stage>.invalid_request`, `Retryable==false`. Why: ARCH-012.
4. **`upstream_5xx_or_network_retryable`** — 503 / `HttpRequestException` (no status) → `<stage>.upstream_unavailable`, `Retryable==true`. Why: ARCH-012.
5. **`timeout_maps_retryable`** — `OperationCanceledException`/timeout → `<stage>.timeout`, `Retryable==true`. Why: ARCH-012 + timeout contract.
6. **`empty_transcript_is_cascade_scoped_retryable`** — empty-STT case → `Code=="cascade.empty_transcript"` (NOT `<stage>.…`), `Retryable==true`. Why: ARCH-012 special case (it's a cascade-level signal, not stage-prefixed).
7. **`unknown_fallback_not_retryable`** — unrecognized → `<stage>.unknown`, `Retryable==false`. Why: ARCH-012 fallback.
8. **`stage_token_interpolates`** — same condition with stage `translation`/`tts` → `translation.timeout`/`tts.rate_limited` etc. Why: the `<stage>` token is parameterized.
9. **`safeMessage_carries_no_secret_or_stack`** — given an exception whose message embeds an api-key-like substring + a stack, the produced `SafeMessage` contains neither (B.1 sets a safe generic message; B.8 is the full sanitizer). Why: ARCH-018/019 defense-in-depth at the boundary.
