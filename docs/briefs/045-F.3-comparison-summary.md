# /tdd brief — comparison_summary (F.3)

## Feature
The **`ComparisonSummary`** view (Flow E): aggregate a session's measured evidence into a Realtime-vs-Cascade comparison — **avg latency by mode**, **estimated cost/min by mode AND by model variant** (both realtime models + both translation models), **error counts**, **WER summary**, **turn counts**. Per-mode aggregates + WER come from `GET /api/sessions/{id}/summary` (already mirrored); the per-model-variant cost breakdown is **derived frontend-side** from the session's per-turn `cost.model` (a pure aggregation — the §13 "frontend computes what the backend doesn't pre-aggregate" precedent), since `ModeSummary` is per-mode only.

Carry-forward folded in: F.3 is the comparison-summary consumer; it also DECIDES the cascade `playback.started` latency-channel question (carry-forward item, may stay `n/a`).

## Acceptance criteria (what "done" means)
- [ ] `ComparisonSummary` renders a **by-mode** comparison from `GET /summary` (`SessionSummary`): per mode (realtime, cascade) — avg latency (the present `ModeSummary` avg fields), estimated cost/min, error count, turn count; plus the session-level `WerSummary` (sample count + avg WER) and total turn count.
- [ ] `ComparisonSummary` renders a **by-model-variant** estimated-cost/min breakdown — grouped by `(mode, model)` over the session's per-turn `CostEstimate` (`model` + `estimatedUsdPerMinute`), averaging cost/min per variant. Both translation models (`gpt-5.4-nano`/`-mini`) and both realtime models (`gpt-realtime`/`-mini`) appear when used (per Q4).
- [ ] The per-variant aggregation is a **pure, unit-TDD'd util** (no render) — given a list of `{mode, cost:{model, estimatedUsdPerMinute}}` it returns `{mode, model, avgCostPerMinuteUsd, turnCount}[]`; **null/absent cost is skipped** (degrade-don't-fabricate — never a synthetic 0, mirroring the backend null-skipping aggregator + lesson §13/§9); an empty session → an empty breakdown (the panel shows an explicit "no turns yet" state, never a crash or a fake 0).
- [ ] Honest degradation: a missing/`n/a` `ModeSummary` field renders as **"n/a"** (never 0); a mode with no turns is absent or "n/a" (not a fabricated row). The cascade `avgSpeechEnd*` is `n/a` (the documented no-client-latency-channel limitation — see Q5/carry-forward).
- [ ] Data sources (per Q1): per-mode + WER from `sessionsApi.getSummary`; per-variant cost from `sessionsApi.getSession` (the canonical persisted turns, complete for both modes). Both fetched via the existing `http` boundary; errors → the store (single sink, §2/§7).
- [ ] `ComparisonSummary` mounted in `App.tsx` (gated on a session that has run ≥1 turn / has a summary — show a "run some turns to compare" hint otherwise).
- [ ] Component test (jsdom/Testing-Library, §14) + the aggregation util's unit tests pass; `npm run format:check && npm run lint && npm run typecheck && npm run test` clean (format:check FIRST).

## RED test outline (Step 2)

### Feature A — `comparisonAggregation` (pure util, §13)
1. **`groups_cost_by_mode_and_model`** — turns across 2 modes × 2 models → one `{mode, model, avgCostPerMinuteUsd, turnCount}` row per `(mode, model)` present; the average is correct. Why: the core per-variant aggregation.
2. **`averages_cost_per_minute_per_variant`** — 2 cascade `gpt-5.4-nano` turns at 0.10 + 0.20 → avg 0.15. Why: cost/min is averaged per variant (ARCH-014).
3. **`skips_null_cost_never_fabricates_zero`** — a turn with `cost: null`/absent `estimatedUsdPerMinute` is EXCLUDED (not counted as 0); a variant with all-null cost → that variant absent (not a 0 row). Why: degrade-don't-fabricate (§9/§13; a synthetic 0 reads as "free").
4. **`empty_session_empty_breakdown`** — no turns → `[]` (the panel shows a "no turns" state). Why: no crash / no fake row.
5. **`distinguishes_realtime_variants`** — realtime turns with `cost.model` `gpt-realtime` vs `gpt-realtime-mini` → 2 distinct rows under mode=realtime. Why: the realtime model-variant split (the deliverable; E.2b set `cost.model` to the realtime model).

### Feature B — `loadComparison` flow (DI'd, §7) — if extracted (Q1)
6. **`loads_summary_and_session_and_aggregates`** — `getSummary` + `getSession` resolve → returns `{summary, byVariant}` (the aggregation ran over the session turns). Why: the data path.
7. **`summary_fetch_error_routes_to_store`** — `getSummary` rejects (`ApiError`) → `store.addError`, the panel degrades (no crash). Why: single error sink (§2/§7).
8. **`session_fetch_error_degrades_variant_breakdown`** — `getSession` rejects → the per-mode summary still renders (from `getSummary`); the variant breakdown shows a degraded/"unavailable" state, error → store. Why: the two sources degrade independently (the comparison's headline per-mode metrics survive a per-variant fetch failure).

### Feature C — `ComparisonSummary` (jsdom/Testing-Library, §14)
9. **`renders_by_mode_comparison`** — given a `SessionSummary` (both modes) → renders both modes' avg latency / cost/min / errors / turn counts + the total + the `WerSummary`. Why: Flow E.
10. **`renders_na_for_missing_fields`** — a `ModeSummary` with null `avgSpeechEndToFirstAudioMs` → renders "n/a" (not "0"). Why: honest degradation (the cascade-no-client-latency-channel case).
11. **`renders_by_model_variant_cost`** — given a `byVariant` breakdown → renders the per-variant cost/min rows (mode + model + cost). Why: the deliverable.
12. **`empty_state_when_no_turns`** — no summary / no turns → a "run some turns to compare" state, not a crash or fake numbers. Why: the gate + empty UX.

## Things to flag at Step 2.5
1. **(LOAD-BEARING) The per-model-variant cost data source.** `ModeSummary` is per-mode only (no model breakdown — verified in `SessionSummaryService`), so the per-variant split must come from per-turn `CostEstimate`. Options: **(a)** `GET /session` → the canonical persisted turns (each `InterpretationTurn` has `CostEstimate` for BOTH modes — cascade WS-priced, realtime `/complete`-priced; complete + survives a fresh load); **(b)** the live `UiSessionState.turns` (`TurnViewModel.cost`) — convenient but **may MISS realtime cost** (realtime cost is computed backend-side at `/complete`; the frontend turn machine may not write it back) + is empty on a fresh load. My default vote: **(a) `GET /session`** — it's the canonical, both-modes-complete source; the per-mode aggregates still come from `GET /summary`. Each source owns what it's authoritative for (no double-truth): `/summary` = per-mode + WER; `/session` turns = the per-variant split. **Confirm realtime turns carry `CostEstimate` in the persisted session** (E.2b prices realtime at `/complete` → persisted via `FinalizeTurn`; degraded/unpriced → null, correctly skipped).
2. **Where the aggregation lives.** Default: a NEW pure `comparisonAggregation.ts` util (unit-TDD'd, §13) rather than `selectors.ts` (keeps `selectors.ts` focused; the aggregation is comparison-specific). My default vote: **a dedicated util.**
3. **Reading the wire turn — a focused projection, not a full graduation.** The wire `InterpretationTurn` is intentionally opaque (`Record<string, unknown>`). Default: define a minimal `ComparisonTurn = { mode: InterpretationMode; cost?: CostEstimate }` projection (read only what the aggregation needs) rather than graduating the whole turn shape. My default vote: **a focused projection** (minimal cross-doc surface; the opaque turn stays opaque per the pragmatic-accrete row).
4. **Unused model variants — show with `n/a`/0-turns, or omit?** The PRD/demo runs both translation models + (optionally) both realtime models, so both should appear after the demo script. Default: **show only model variants that have ≥1 turn** (the comparison reflects real measured data; don't fabricate rows for models never run — a fabricated 0-turn row is misleading). The demo script ensures the compared variants are exercised. My default vote: **only-used variants** (+ a note that the comparison reflects what was run). Flag if you'd rather show all catalog models with explicit "not run".
5. **Cascade `playback.started` / speechEnd→playback = `n/a` (carry-forward decision).** The cascade WS has no client→server latency-report channel, so the persisted/aggregated cascade `avgSpeechEnd*` is `n/a` (honest degradation — D.5/D.6). F.3 is the last-consumer of this carry-forward: **decide whether to add a cascade client→server latency channel for the comparison, or accept `n/a` + document.** My default vote: **accept `n/a` + document in the G.5 limitations** (adding a channel is a cross-area cascade-WS change — a backend+frontend slice — out of scope for this frontend comparison view; the comparison still has the backend-measured stage latencies + the realtime client-timing). If you think the missing cascade end-to-end latency materially weakens the comparison deliverable, flag it and I'll escalate the add-a-channel decision.
