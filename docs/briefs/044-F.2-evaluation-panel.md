# /tdd brief — evaluation_panel (F.2)

## Feature
The standalone **`EvaluationPanel`** (Flow D): select a scripted phrase → read it aloud → record → `POST /transcribe` (STT-only hypothesis) → `POST /wer` → display + **persist** the `WerResult` on the session (so F.3's comparison summary can aggregate it). Consumes the three landed F.1 endpoints via a new `evaluationApi` + a DI'd `evaluationActions` flow; renders the "WER is STT-only" explanation.

## RED test outline (Step 2)

### Feature A — `evaluationApi` (the client)
1. **`get_phrases_requests_and_parses`** — `getPhrases()` GETs `/api/evaluation/phrases`, returns the parsed `EvaluationPhrase[]`. Why: ARCH-009 contract + lesson §3/§4 (fresh `Response` per call).
2. **`transcribe_posts_multipart_no_content_type`** — `transcribe(params, blob)` POSTs `/api/evaluation/transcribe` as `FormData` (sessionId/phraseId/language/audio set), **no `Content-Type` header** (fetch sets the boundary — assert the header is absent, lesson §3 multipart). Returns the parsed `TranscribeResponse`.
3. **`compute_wer_posts_json`** — `computeWer(req)` POSTs `/api/evaluation/wer` as JSON, returns the parsed `WerResponse`. Why: ARCH-009.
4. **`api_error_propagates_as_ApiError`** — a non-OK status → the `http` boundary throws `ApiError(UiError)` (not a raw body). Why: lesson §3 (single failure boundary).

### Feature B — `evaluationActions` (the Flow-D flow, DI'd)
5. **`run_evaluation_happy_path`** — record → transcribe → (Q1 turn) → wer → returns/exposes the `WerResult`; the store has NO error. Asserts the call ordering (record before transcribe before wer) + that `computeWer` is called WITH a `turnId` (per Q1). Why: Flow D + the persist requirement.
6. **`run_evaluation_persists_with_turn_id`** — asserts the `WerRequest` carries a non-empty `turnId` (the persist path; the F.3 dependency). Why: AC "WER is persisted."
7. **`run_evaluation_persistence_warning_surfaced`** — a `WerResponse.persistenceWarning` → `store.addError(warning)` (surfaced, not thrown). Why: mirrors `endSession` (degrade-don't-crash).
8. **`run_evaluation_transcribe_error_routes_to_store`** — a `transcribe` rejection (`ApiError`) → `store.addError(uiError)`, the flow aborts before `wer`. Why: single error sink (§2/§7); don't score a failed transcribe.
9. **`run_evaluation_non_api_error_fallback_no_leak`** — a non-`ApiError` throw → a typed fallback `UiError` (no raw message leak). Why: §7 (pin both error arms).
10. **`run_evaluation_mic_null_aborts`** — `recordBlob()` resolves `null` (mic denied / capture failed) → the flow aborts, no transcribe/wer call, the capture error is already surfaced. Why: lesson §8/§11 (mic-denied abort).

### Feature C — `EvaluationPanel` (jsdom/Testing-Library, §14)
11. **`renders_phrase_selector_and_reference`** — phrases load → the selector + the selected reference text render. Why: Flow D UI.
12. **`renders_wer_explanation_verbatim`** — the "WER is STT-only" explanation string is present. Why: ARCH-015 UI explanation (spec-required copy).
13. **`record_button_dispatches_flow`** — clicking record invokes the injected/dispatched evaluation flow (spy). Why: reachability — the panel is wired to the action, not dead.
14. **`renders_wer_result`** — given a computed `WerResult` (store/local), the panel shows the WER % + S/I/D/N + the hypothesis. Why: Flow D result display.

## Things to flag at Step 2.5
1. **(LOAD-BEARING) How does the panel get a `turnId` to persist WER?** `POST /wer` persists only when a `turnId` is supplied (server-authoritative); the `transcribe` endpoint is stateless (no turn). Options: **(a)** create a DEDICATED evaluation turn per measurement (`sessionsApi.createTurn(sessionId)` → use its `turnId`) — the `WerResult` attaches there + F.3's `WerSummary` aggregates it; **(b)** attach to the session's most-recent interpretation turn (no new turn) — but conflates eval WER with that turn + requires a prior turn (breaks "standalone"); **(c)** compute-only, no persist — rejected (F.3's `WerSummary` stays empty; AC violated). My default vote: **(a) dedicated evaluation turn.** **KNOWN LIMITATION it introduces (verified against `SessionSummaryService`): `ModeSummary.TurnCount` counts ALL turns of a mode, so a dedicated eval turn inflates that mode's turn count in F.3's comparison.** The quality metrics are UNAFFECTED — latency/cost averages are null-skipping (an eval turn has no latency/cost → skipped), and `WerSummary` is exactly right. So the only skew is the per-mode `turnCount`. **Default: ship (a) + document the limitation in the G.5 write-up** ("turn counts include standalone WER-evaluation turns"). A clean fix (a backend eval-turn marker so `ModeSummary` excludes them) is a later backend slice — out of scope for this frontend slice. _(I'm flagging this to the lead as a deliverable-touching heads-up; if the user wants the backend exclusion instead of documentation, that re-sequences as a backend slice — but it doesn't block F.2's frontend work either way: the eval turn still gets created; the backend later learns to exclude it.)_
2. **Where does eval UI state live — local panel state or a store slice?** Default: **phrases / selectedPhraseId / hypothesis / werResult / eval-status live as LOCAL panel state** (the panel is standalone + self-contained), while **errors always route to the store** (the single error sink, lesson §2) and the flow lives in the DI'd `evaluationActions` module (lesson §7). My default vote: **local state + store-as-error-sink** — avoids growing `UiSessionState` for a standalone panel. If you find the WER needs to survive a remount or be read elsewhere (it doesn't for F.2; F.3 reads the persisted summary, not this transient), revisit.
3. **Audio capture — reuse `audioCaptureController.recordBlob()`?** It exists (`Promise<BlobCapture {blob, mimeType} | null>`, probes the MediaRecorder mime per ARCH-030). Default: **reuse it** (don't add a second capture path); send `blob` + let the content-type ride (the backend `CascadeUploadValidation` accepts the MediaRecorder container + strips MIME params). My default vote: **reuse `recordBlob`.** Fixed duration vs click-to-stop: default a fixed `recordBlob(durationMs)` (simpler for a scripted phrase) — flag if you prefer a manual stop.
4. **TS mirror shapes — confirm against Appendix A.** `EvaluationPhrase {phraseId, language, referenceText, category}`; `WerResult {phraseId, reference, hypothesis, normalizedReference, normalizedHypothesis, substitutions, insertions, deletions, referenceWordCount, wer}`; `TranscribeResponse {hypothesis, sttProvider, sttModel, latencyEvents}`; `WerRequest {sessionId, turnId?, phraseId, hypothesis}`; `WerResponse {result: WerResult, persistenceWarning?: UiError}`. My default vote: **mirror exactly as above** (camelCase; `language` is a `LanguageCode` union; `latencyEvents` is the existing `LatencyEvent[]` mirror). Confirm no drift vs the backend records before GREEN.
5. **Phrase fetch timing + config-gating.** Default: fetch phrases on panel mount (lazy; the panel is standalone) and gate the panel on a live session (it needs the `turnId`/persist path + a `sessionId`). My default vote: **fetch-on-mount, gate-on-active-session** (show a "start a session to evaluate" hint when not active, mirroring the other panels' gating).
