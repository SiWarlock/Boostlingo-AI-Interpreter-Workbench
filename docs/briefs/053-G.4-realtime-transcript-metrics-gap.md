## Feature
Make realtime turns render **source + target transcripts** and produce **metrics/summary** (currently only audio plays). Pin which of three gaps are real — (A) transcript GA-event mapping/accumulation, (B) input-transcription not enabled in the session config, (C) latency/turn events not reported to the backend — fix the frontend parts, flag any backend part.

## Investigate + fix (pin which gaps are real)

### (A) Transcript GA-event mapping → store (frontend — `realtimeEventSink.ts`)
- Is **`response.output_audio_transcript.delta`** (the TARGET transcript) mapped + **accumulated** into the store's target transcript? (Incremental deltas → cumulative, per lesson §16's accumulate pattern.)
- Is **`conversation.item.input_audio_transcription.delta` / `.completed`** (the SOURCE transcript) mapped + accumulated into the source transcript?
- Confirm the exact GA `type` strings against the live event stream (the §18/§24 verify-the-real-shape discipline; the carry-forward flagged these as smoke-confirm items — ARCH-010 §7). If the sink ignores/mis-keys these events, that's gap A → fix the mapping + accumulation. TDD the mapping (the sink is unit-testable — feed a fake event, assert the store accumulates).

### (B) Is input transcription ENABLED? (session config — frontend session.update AND/OR backend broker mint)
- Where is the realtime session config set: the client's `session.update` (frontend — lesson §17 sends `turn_detection:null` there) and/or the E.1 broker `client_secrets` mint (backend `RealtimeController`)? 
- Does it set **`audio.input.transcription` / `input_audio_transcription` (model e.g. `gpt-4o-transcribe`, ARCH-010)**? If input transcription is OFF, the SOURCE transcript never arrives (gap B).
  - If it's settable via the client `session.update` → **fix frontend** (add it).
  - If it must be set at the broker mint → **FLAG (backend)**; I route a small backend broker-config slice (don't fix backend from here).
  - If input transcription genuinely can't be enabled on this config → render the **"source transcript unavailable" note** (ARCH-010, the existing TranscriptPanel branch) — but the user EXPECTS transcripts, so enabling it is the goal; the note is the fallback.
- Also confirm the client requests the OUTPUT transcript (so `output_audio_transcript.delta` arrives).

### (C) Metrics/summary — events reported to the backend? (frontend — sink + turn lifecycle)
- Realtime metrics come from events reported via **`appendTurnEvents` (POST /turns/{id}/events)** → backend aggregation → the summary (lesson §16's "report client events so the backend aggregates the canonical realtime metrics"). If only `audioDelta` is handled and the latency/transcript events aren't reported, OR the turn doesn't complete/persist, the summary stays "Realtime: n/a."
- Confirm: the sink stamps + **reports** the latency events (first-audio, recording stamps); the turn **completes** (`responseDone` → `completeTurn`) + the events post via `appendTurnEvents`; the backend aggregates → `GET /summary` returns realtime metrics. Fix whichever link is broken.
- **Cost (related):** realtime cost is `/complete`-priced (E.2b); the user's "cost missing" for realtime is downstream of the turn completing/persisting — verify it populates once the turn completes (likely resolved by C; if not, flag).
