# /tdd brief — global_exception_handler (B.9a) — SAFETY

## Feature
A **global sanitizing exception handler** — an `IExceptionHandler` that catches any otherwise-unhandled exception at the HTTP boundary, routes it through B.8's `ErrorSanitizer`, and writes a safe `UiError` JSON response with a safe status code. **No stack trace, secret, or raw payload reaches the client** (the original is logged server-side by the sanitizer). Wired into the request pipeline via `app.UseExceptionHandler()`. This is the **boundary application of safety invariant #4** — without it, an unhandled exception bypasses the sanitizer entirely.

> **B.9 is split into three commits for safety-isolation** (two distinct safety invariants can't share a commit + the controller is large): **B.9a** (this brief — global exception handler, invariant #4 boundary), **B.9b** (ConfigController key-presence/no-secret-echo, invariant #1), **B.9c** (SessionsController CRUD + `Result`→DTO mapping). This brief is B.9a only.

## Use case + traceability
- **Task ID:** B.9 (part a)
- **Architecture sections it implements:**
  - `ARCH-019` — security boundary #4: provider/internal errors are sanitized (no stack traces / no secrets) before UI display; **a default ASP.NET error response (which leaks a stack in Development) must never reach the client.**
  - `ARCH-018` — normalized error model; the response carries a `UiError`, not a framework error page / `ProblemDetails`.
  - `ARCH-009` — the error response body is the `UiError` shape `{ code, safeMessage, stage?, retryable, turnId? }`.
- **Safety invariant pinned:** #4 — no stack/secret/raw payload crosses the HTTP boundary, even on an *unhandled* exception path.
- **Related context:**
  - **Reuses B.8 `ErrorSanitizer.Sanitize(Exception)`** (registered DI singleton) — the handler is **thin**: it does NOT re-implement sanitization or re-log; it routes the exception through the sanitizer and serializes the resulting `UiError`. (Lesson §13 — safe-by-construction; one sanitizer owner.)
  - The sanitizer already logs the original server-side (single-lined), so the handler does not double-log (→ 2.5 Q5).
  - HTTP status comes from the sanitizer's server-only `[JsonIgnore] UiError.HttpStatusCode` (`?? 500`); the body is the exact TS-mirror `UiError` (the `HttpStatusCode` field is NOT serialized).
  - `Program.cs` is minimal-API today (only `GET /api/health`); controllers arrive in B.9b/B.9c. The handler is **middleware** — it works for both minimal-API + controller endpoints, so it needs no controller infrastructure.
  - Consumed-by / protects: every current + future endpoint (health, B.9b config, B.9c sessions, F.1 evaluation). C.4's WS path emits `error` frames directly (not via this HTTP handler).

## Acceptance criteria (what "done" means)
- [ ] A `GlobalExceptionHandler : IExceptionHandler` whose `TryHandleAsync(HttpContext, Exception, CancellationToken)` routes the exception through `ErrorSanitizer.Sanitize`, sets `response.StatusCode` from the resulting `UiError.HttpStatusCode` (`?? 500`), writes the `UiError` as camelCase JSON (via `JsonDefaults`), sets `Content-Type: application/json`, and returns `true` (handled).
- [ ] **Safety pin:** an exception whose message/stack contains an API-key substring (`sk-…`/`ek_…`) + stack frames → the response body contains **neither** the secret **nor** stack text (scan the full serialized body); `code = internal.error`, `retryable = false`.
- [ ] The response body deserializes to the `UiError` shape (no `provider`, no `httpStatusCode` field on the wire, no `exception`/`stackTrace`/`detail` keys).
- [ ] Status is `500` for a generic exception (the sanitizer's default); if the sanitizer resolves a different status it is honored.
- [ ] Wired in `Program.cs`: `builder.Services.AddExceptionHandler<GlobalExceptionHandler>()` + `app.UseExceptionHandler()` placed so it catches exceptions from all endpoints (→ 2.5 Q6).
- [ ] All unit tests in `AiInterpreter.Tests/GlobalExceptionHandlerTests.cs` pass. `/preflight` clean. **Mandatory `security-reviewer` pass at Step 8.**

## Things to flag at Step 2.5
1. **Handler mechanism.** My default vote: the **.NET 8 `IExceptionHandler`** (`AddExceptionHandler<T>()` + `app.UseExceptionHandler()`) — DI-injectable (takes `ErrorSanitizer`), directly unit-testable, the modern idiom. Note `app.UseExceptionHandler()` with an `IExceptionHandler` may require `AddProblemDetails()` registered for the fallback path — include it if the wiring needs it, but **our handler writes a `UiError`, never `ProblemDetails`**. Alternative: an inline `app.UseExceptionHandler(appBuilder => appBuilder.Run(...))` lambda. Flag if you prefer the lambda.
2. **Output is `UiError`, not `ProblemDetails`.** My default vote: the body is always the `UiError` shape (ARCH-009 — the frontend `ErrorBanner` consumes `UiError`); do NOT emit RFC7807 `ProblemDetails`. Confirm.
3. **Status source.** My default vote: `response.StatusCode = uiError.HttpStatusCode ?? 500` (the sanitizer's server-only field; generic exception → 500). 
4. **Test strategy.** My default vote: **unit-test the handler class** (`DefaultHttpContext` + `MemoryStream` body) — strong, no production test-route pollution; add the WebApplicationFactory integration test only if cheap (test route via the test host builder, never in `Program.cs`).
5. **Logging.** My default vote: **rely on the sanitizer's server-side log** (B.8 already logs the original single-lined); the handler does NOT re-log (avoid duplicate log lines). Flag if you'd rather the handler own the log.
6. **Pipeline placement.** My default vote: `app.UseExceptionHandler()` early (right after `var app = builder.Build();`, before `UseSwagger`/`UseWebSockets`/`UseCors`/endpoints) so it wraps the whole pipeline. Confirm.
