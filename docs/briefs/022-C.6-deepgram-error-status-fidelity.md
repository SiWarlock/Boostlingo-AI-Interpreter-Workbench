# /tdd brief — deepgram_error_status_fidelity

## Feature
Restore Deepgram error-status fidelity: extract the HTTP status from the Deepgram SDK exception (`DeepgramException`/`DeepgramRESTException`, which carries no `HttpStatusCode` property) so a common-path **429/401/403/400/5xx** maps to the correct `ProviderError` (rate-limited-retryable / auth-non-retryable / invalid-request / upstream-unavailable) instead of degrading to `stt.unknown`. Closes the C.1 Q5 gap — **user pulled this forward (option ii)** because error fidelity feeds the G.5 comparison write-up.

## Use case + traceability
- **Task ID:** C.6 (Q5 item — the error-status fidelity sub-item; the unbounded-channel + Connect/Stop-race C.6 items stay deferred/lower-priority).
- **Architecture sections it implements:** `ARCH-012` (§9 exception→`ProviderError` mapping table + `ProviderErrorMapper`), `ARCH-018` (normalized error codes / retryability).
- **Related context:**
  - **Origin:** C.1 Step-9 Q5 finding (brief 020; lesson §18 "SDK-fidelity caveat"). C.1 shipped `3e5e899` with the honest `stt.unknown` degrade documented as intended + test 8 pinning it. This slice flips that.
  - **Cross-doc invariant change (the user/lead flagged this is a bounded ARCH-012 mapper change):** the **mapping-table semantics are unchanged** (same status→code map). What changes is that the **Deepgram path now reaches the table** by extracting status from the SDK exception, plus (likely) a new public mapper entry point. Flag at Step 9; orchestrator writes the ARCH-012 note + Appendix A row.
  - **Keep `ProviderErrorMapper` vendor-agnostic** — it lives in `Providers/Abstractions` and must NOT take a dependency on the Deepgram SDK. The Deepgram-specific *extraction* lives in the Deepgram provider/mapping; it calls the mapper with an explicit status.

## Acceptance criteria (what "done" means)
- [ ] A Deepgram SDK exception representing **429** → `SttFailed{Code="stt.rate_limited", Retryable=true}`; **401/403** → `stt.auth` (non-retryable); **400** → `stt.invalid_request` (non-retryable). A genuinely **unmappable** exception (no recoverable status) → `stt.unknown` (the existing fallback survives). _(**Step-1 reality, accepted residual:** the SDK's `ErrCode` is the Deepgram **`err_code` semantic string** — `"TOO_MANY_REQUESTS"`/`"INVALID_AUTH"`/`"INSUFFICIENT_PERMISSIONS"`(401-perms+403, both→auth)/`"Bad Request"` — matched string→status, NOT `int.Parse`. **5xx-with-body degrades to `stt.unknown`** because Deepgram 5xx bodies use a different JSON key (`error_code` ≠ `err_code`) → `ErrCode` stays at the SDK default (`"Unknown Error Code"`, verified by reflection — not null); matching the non-contractual `ErrMsg` is rejected as fragile. 5xx-with-**empty**-body still recovers `upstream_unavailable` via the existing `HttpRequestException`→`Map` path. The user's stated goal (429/401/403 + auth) is fully met.)_
- [ ] `ProviderErrorMapper` stays **vendor-agnostic** — no Deepgram-SDK reference in `Providers/Abstractions/`. The Deepgram exception→status extraction lives in `DeepgramSttMapping` (or the Deepgram provider) and calls the mapper with the extracted status.
- [ ] **The C.1 test-8 expectation flips:** the existing `deepgram_exception_without_status_yields_unknown` either (a) becomes the genuinely-unmappable-still-unknown case, or (b) is updated to the new status-bearing expectation — whichever keeps both behaviors pinned (status-recovered → mapped; truly-unmappable → unknown). No silently-deleted coverage.
- [ ] The ARCH-012 status→code map is applied unchanged (reuse the existing `MapHttp`/table logic — do NOT duplicate the table; lesson §17 / §5).
- [ ] Full suite green; `/preflight` clean. Cross-doc invariant flagged at Step 9 (orchestrator writes ARCH-012 + Appendix A).

## Things to flag at Step 2.5
1. **[LOAD-BEARING] What does the Deepgram SDK exception actually carry? VERIFY at Step 1.** Is the HTTP status in `ErrCode` (numeric/string), a dedicated field, or only parseable from `ErrMsg`? **Default vote: if `ErrCode` carries the HTTP status code, parse it → `int` → mapper.** If `ErrCode` is an opaque Deepgram identifier (not the HTTP status), map the **known-important** ones (rate-limit, auth) → status/code and fall back to `stt.unknown` for the rest — and **flag the partial coverage** (the user wants 429/401/403 correct; full coverage of every Deepgram code is not required). The whole slice's shape depends on this — confirm before designing the extraction.
2. **Mapper entry point.** (a) Add public `ProviderErrorMapper.MapStatus(int statusCode, provider, stage)` and refactor the private `MapHttp` to delegate; Deepgram calls `MapStatus(status, "deepgram", "stt")`. (b) Construct a synthetic `HttpRequestException` with the status and pass it to the existing `Map`. **Default vote: (a) `MapStatus`** — cleaner than fabricating an exception, keeps the table single-owned + vendor-agnostic, and is the honest cross-doc surface change. (This is the cross-doc flag.)
3. **Extraction location.** Extend the existing `DeepgramSttMapping.ToFailed` (already owns Deepgram exception→`SttFailed`) vs a new helper. **Default vote: extend `ToFailed`** — co-located, already tested, minimal surface.
4. **Scope — Deepgram-only.** C.2/C.3 (raw `HttpClient`) surface status-bearing errors and don't need this. **Default vote: Deepgram-STT-specific extraction; do NOT generalize.** The `MapStatus` entry point is reusable, but the extraction is Deepgram's.
