# /tdd brief — openai_tts_provider

## Feature
Implement the real `OpenAiTtsProvider` (`ITtsProvider`) behind the unchanged B.1 interface: a **chunk-transfer streamed** OpenAI `POST /v1/audio/speech` call that emits `TtsStarted → TtsFirstAudio → TtsAudioChunk* → TtsComplete` from the streamed audio body (first chunk = `tts.first_audio`, never a relabeled completion), guards the 4096-char input cap as a non-retryable error **before** the call, and maps HTTP/exception failures to `ProviderError` via `ProviderErrorMapper`. Third real provider of Phase C; reuses the C.2 raw-`HttpClient` pattern (lesson §20) but over a **binary chunk stream**, not SSE.

## Use case + traceability
- **Task ID:** C.3
- **Architecture sections it implements:** `ARCH-011` (§8 TTS path — chunk-streamed `/v1/audio/speech`), `ARCH-012` (§9 `ITtsProvider` contract, exception→`ProviderError`, `OpenAiTtsOptions`).
- **Related context:**
  - **Reuses lesson §20 (raw-`HttpClient` provider):** thin transport shell + pure unit-TDD'd mapper; mock `HttpMessageHandler` returns a **canned binary chunked body** → TDD the streaming end-to-end; the per-stage **timeout must cover the whole read loop** (not just the send); the **OCE-filter polarity** with a trailing generic catch is `catch (OCE) when (ct.IsCancellationRequested) { throw; }` + generic-arm-maps-timeout (do NOT invert — §20). Reuse `ProviderErrorMapper.Map(ex, "openai", "tts")` (status-bearing — no C.1 Q5 gap); reuse C.1's `InternalsVisibleTo`.
  - **Match the fake contract** (`FakeTtsProvider`, B.2): `TtsStarted → TtsFirstAudio(ContentType) → TtsAudioChunk(Bytes, Seq)* → TtsComplete(ContentType)`; the no-chunk variant (`TtsFirstAudio → TtsComplete`) is conformant (B.10 success contract is tolerant of it).
  - **KEY DIFFERENCE vs C.2:** the response body is a **streamed binary audio** (mp3 chunks), **not SSE text** — read raw byte buffers, no `data:`/event parsing. `ContentType` comes from the response header.
  - **Carry-forward (origin C.2):** if reusing the in-band-error pattern, consider a shared in-band-error→`ProviderError` factory (C.2 used a fixed `InvalidOperationException` discriminator). For TTS, errors are HTTP-status (pre-stream) — likely no in-band-error needed; confirm at Step 1.

## Acceptance criteria (what "done" means)
- [ ] `OpenAiTtsProvider : ITtsProvider` exists; `SynthesizeAsync` streams `/v1/audio/speech` and yields `TtsStarted` → `TtsFirstAudio` (on the genuine first audio chunk) → `TtsAudioChunk(bytes, seq)*` → `TtsComplete` — **genuinely streaming, first-audio is the real first chunk, never a relabeled completion** (forbidden #3; ARCH-011 "`tts.first_audio` is the first chunk").
- [ ] **`TtsAudioChunk.Seq`** increments per emitted chunk (start value confirmed at Step 2.5); `TtsFirstAudio.ContentType` + `TtsComplete.ContentType` come from the response `Content-Type` header (e.g. `audio/mpeg` for mp3).
- [ ] **4096-char input cap (ARCH-011):** if `request.Text.Length > 4096`, emit `TtsFailed(tts.invalid_request, Retryable=false)` **without making the HTTP call** (assert no request is sent). Non-retryable.
- [ ] **Request body** sets `model`, `voice` (from `request.Voice`), `input` (the text), `response_format` (from `request.ResponseFormat`/options, default `mp3`), and streaming per the verified API shape; optional `instructions` when supplied.
- [ ] **Error mapping:** HTTP/exception → `TtsFailed(ProviderErrorMapper.Map(ex, "openai", "tts"))`; `429→tts.rate_limited`(retryable), `401/403→tts.auth`(non-retryable). No new mapper codes.
- [ ] **Timeout covers the whole read loop** (linked CTS scoped to stream-open + each chunk read → stalled stream = `tts.timeout`); OCE handling per §20.
- [ ] **No leak:** the OpenAI API key flows only from `OpenAiTtsOptions` (server-side) into the `Authorization` header; never into any `TtsFailed`/`ProviderError`/log.
- [ ] All unit tests in `AiInterpreter.Tests/OpenAiTtsProviderTests.cs` pass; full suite stays green (185 + new).
- [ ] `/preflight` clean.

## RED test outline (Step 2 — `AiInterpreter.Tests/OpenAiTtsProviderTests.cs`)
Drive via a mock `HttpMessageHandler` returning a canned **binary** chunked response (with a `Content-Type` header).

1. **`streams_started_firstaudio_chunks_complete_in_order`** — a canned chunked audio body → ordered `[TtsStarted, TtsFirstAudio(ct), TtsAudioChunk(seq0), TtsAudioChunk(seq1), …, TtsComplete(ct)]`. Why: ARCH-011 streaming; proves genuine chunk streaming.
2. **`first_audio_is_the_first_chunk`** — `TtsFirstAudio` precedes the first `TtsAudioChunk` and is stamped on the real first chunk arrival, not relabeled from completion. Why: ARCH-011/013 `tts.first_audio` honesty; forbidden #3.
3. **`content_type_from_response_header`** — the response `Content-Type: audio/mpeg` → `TtsFirstAudio.ContentType`/`TtsComplete.ContentType == "audio/mpeg"`. Why: real content-type, not assumed.
4. **`input_over_4096_chars_fails_without_call`** — `request.Text.Length > 4096` → `TtsFailed{Code="tts.invalid_request", Retryable=false}` and **no HTTP request is sent** (assert via the mock handler's call count). Why: ARCH-011 input cap; pre-call guard.
5. **`rate_limited_yields_failed_retryable`** — mock 429 → `TtsFailed{Code="tts.rate_limited", Retryable=true, Stage="tts", Provider="openai"}`. Why: ARCH-012 via mapper (status-bearing — no Q5 gap).
6. **`auth_error_yields_failed_nonretryable`** — mock 401/403 → `TtsFailed{Code="tts.auth", Retryable=false}`. Why: ARCH-012 table.
7. **`request_body_sets_voice_format_input`** — capture the outgoing request → assert `model`, `voice` (from `request.Voice`), `input`, `response_format` (default `mp3`). Why: ARCH-011 config-driven voice/format; a behavioral pin.

7 tests — lean. Reuse the §20 mock-handler harness. Do NOT re-test the mapper truth table (B.1) or `FakeTts` per-variant behavior (B.2) — lesson §17. The whole-read-loop timeout + OCE handling are §20-established (no new test unless the chunk-read path differs materially from C.2's line-read).

## Things to flag at Step 2.5
Defaults pre-loaded. **Q1 is the main fork.**

1. **Chunk-read seam + mapper split.** The binary chunk-read loop (read response stream into buffers → emit `TtsAudioChunk`) lives in the provider shell; the pure `internal static OpenAiTtsMapping` holds `BuildRequestBody` + the 4096-cap guard + `ToFailed`. **Default vote: shell holds the chunk loop (transport), mapper holds request/guard/error.** Tested end-to-end via the mock handler (the chunk loop is thin). Confirm: chunk buffer size (e.g. read whatever the stream gives, or a fixed ~4–16KB buffer), and `Seq` start value (0 vs 1).
2. **`ContentType` source + fallback.** Read from the response `Content-Type` header; if absent, derive from `request.ResponseFormat` (`mp3→audio/mpeg`, `wav→audio/wav`, `pcm→audio/pcm`). **Default vote: header-first, format-map fallback.** (The C.4 serialization boundary will clamp/validate it — not C.3's job.)
3. **4096-cap: exact limit + code.** **Default vote: `> 4096` chars → `tts.invalid_request` (non-retryable), pre-call, no HTTP request.** Confirm the cap is characters (per ARCH-011/OpenAI) and the code (`invalid_request` matches the ARCH-012 400/422 row — a client-side input violation).
4. **Voice resolution.** Provider uses `request.Voice` as-is (the orchestrator/options resolves any `VoiceByLanguage` map upstream) vs the provider resolves the map. **Default vote: provider uses `request.Voice` as-is** — keeps the provider dumb; map resolution is upstream (confirm where `VoiceByLanguage` is applied — likely C.4/options, not here).
5. **In-band errors?** Unlike SSE translation, `/v1/audio/speech` errors are HTTP-status (pre-stream), so likely no in-band-error path. **Default vote: no in-band-error handling needed** (status-bearing only) — confirm at Step 1; if the API can emit a mid-stream error, handle per §20 + the carry-forward shared-factory note.
