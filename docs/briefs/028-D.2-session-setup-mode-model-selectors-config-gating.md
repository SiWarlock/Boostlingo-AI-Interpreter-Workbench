# /tdd brief — session_setup_mode_model_selectors_config_gating

## Feature
Build the session-setup surface: `SessionSetup` (label + direction + realtime-model + translation-model selectors, Start/End) and `ModeToggle` (Realtime/Cascade), both rendering from `sessionStore` and gated by `providerHealth` — unconfigured modes disabled (Flow A), `ModeToggle` disabled during recording/processing/playing. Wires the Start button through `sessionsApi.createSession` → `store.sessionStarted` (closing a D.1 foundation-ahead-of-callers gap).

## Acceptance criteria (what "done" means)
- [ ] A pure gating module (`web/src/state/selectors.ts` — see Step-2.5 Q1) exports: `modeAvailability(config)` → `{ realtime, cascade }`; `availableModels(config)` → `{ realtimeModels, translationModels }`; `canToggleMode(turnStatus)` → `boolean`. All pure, total over `undefined` config.
- [ ] **`availableModels` reads the catalog arrays DIRECTLY** (`config.realtime.models` + `config.cascade.translation.models`) — **NOT gated by `configured`**. The backend always populates the catalogs regardless of key-presence (verified `server/.../Config/ConfigService.cs:48–55` — `Models: [.. RealtimeModels]` etc.); `configured` gates the MODE (`modeAvailability`), not the model list. The only empty case is `undefined` config (not yet loaded). (`CreateSessionRequest` requires both models, so an empty selector when a key is merely absent would break session-create.)
- [ ] `sessionStore` gains `updateSessionConfig(patch)` (see Step-2.5 Q2): merges a partial `{ label?, mode?, direction?, realtimeModel?, translationModel? }`, transitions `idle → configured`, new state ref, notifies subscribers.
- [ ] `ModeToggle` renders Realtime/Cascade from the store's `mode`; a mode is **disabled** when `!modeAvailability(providerHealth)[mode]` OR `!canToggleMode(turnStatus)`; selecting an enabled mode dispatches `updateSessionConfig({ mode })`. (Full Flow-G realtime teardown/re-mint is Phase E — D.2 only enforces the disable guards + the mode write.)
- [ ] `SessionSetup` renders: a label input, a direction selector (en→es / es→en), a realtime-model selector (options = `availableModels(providerHealth).realtimeModels`), a translation-model selector (options = `availableModels(...).translationModels`), and Start/End buttons. Selector changes dispatch `updateSessionConfig(...)`. Selectors constrain to the published catalog (client-side mitigation of the "model allowlist" carry-forward).
- [ ] **Start** flow: **`store.clearErrors()`** (clears any prior transient error — e.g. a failed config bootstrap or a prior failed Start — so it doesn't linger past a successful start; this is the D.1 code-quality reviewer's deferred stale-errors item, routed here) → `setSessionStatus('starting')` → build `CreateSessionRequest` from store state → `sessionsApi.createSession(req)` → on success `store.sessionStarted(session)`; on `ApiError` `store.addError(e.uiError)` + revert status to `configured`. **End** flow: `sessionsApi.endSession(sessionId)` → `store.sessionEnded()` (+ `addError` on the `persistenceWarning` if present).
- [ ] `App.tsx` renders `SessionSetup` + `ModeToggle` (mounted from the shell). Clean separation (ARCH-007): both components render only from `useSessionState` + dispatch store intents / call API clients — **no transport-client internals imported**.
- [ ] All unit tests in `web/src/state/selectors.test.ts` + the new `sessionStore` `updateSessionConfig` cases pass.
- [ ] `/preflight` clean (`npm run lint && npm run typecheck && npm run test`).

## RED test outline (Step 2)

### selectors (`web/src/state/selectors.test.ts`)
1. **`mode_availability_both_configured`** — Asserts: `realtime.configured && all-three-cascade-stages.configured` → `{realtime:true, cascade:true}`. Why: Flow A gating.
2. **`mode_availability_realtime_only`** — Asserts: cascade with one stage `configured:false` → `{realtime:true, cascade:false}`. Why: cascade needs the full STT+translation+TTS pipeline (Q3).
3. **`mode_availability_undefined_config`** — Asserts: `undefined` → `{realtime:false, cascade:false}` (not-yet-loaded → nothing enabled). Why: total function + safe pre-bootstrap render.
4. **`available_models_reads_catalog_regardless_of_configured`** — Asserts: a loaded config returns `config.realtime.models` + `config.cascade.translation.models` straight through; `undefined` config → `{[],[]}`; **a config with `realtime.configured:false` but `models:['gpt-realtime','gpt-realtime-mini']` STILL returns those models** (the backend always populates the catalog — `configured` gates the mode, not the list). Why: selectors constrain to the catalog without conflating mode-enablement; `CreateSessionRequest` requires both models even when a mode is disabled.
5. **`can_toggle_mode_blocks_during_active_turn`** — Asserts: `false` for `'recording'|'processing'|'playing'`; `true` for `'ready'|'captured'|'completed'|'failed'`. Why: ARCH-007 ModeToggle disable rule.

### sessionStore (`web/src/state/sessionStore.test.ts` — added cases)
6. **`update_session_config_merges_and_configures`** — Asserts: `updateSessionConfig({mode:'realtime'})` from `idle` merges the field, sets `sessionStatus 'configured'`, returns a new ref, notifies. A second partial merges without clobbering prior fields. Why: store is the live config source of truth (ARCH-007).

### session actions (`web/src/state/sessionActions.test.ts` — see Step-2.5 Q7; mock store + mock `sessionsApi`)
7. **`start_clears_prior_errors_then_creates_and_starts`** — Asserts: with a prior error in the store, `startSession(...)` calls `clearErrors()` FIRST, then `setSessionStatus('starting')`, then `sessionsApi.createSession` with the store-derived `CreateSessionRequest`, then `store.sessionStarted(session)` on success — and the prior error is gone. Why: the D.1 deferred stale-errors fix + the Start orchestration.
8. **`start_on_api_error_adds_error_and_reverts`** — Asserts: when `createSession` throws `ApiError`, `startSession` calls `store.addError(e.uiError)` + reverts `sessionStatus` to `'configured'` (not stuck in `'starting'`). Why: failure path.
9. **`end_session_ends_and_surfaces_persistence_warning`** — Asserts: `endSession(...)` calls `sessionsApi.endSession` → `store.sessionEnded()`, and if the response carries `persistenceWarning`, `store.addError(warning)`. Why: End flow + ARCH-016 best-effort-persist surfacing.

## Things to flag at Step 2.5
1. **Gating-helper location/name** — `web/src/state/selectors.ts` (pure derivations over `UiSessionState` + `ConfigResponse`; D.6 panels reuse) vs a `components/`-adjacent util. Default vote: **`state/selectors.ts`** — pure, store-adjacent, reusable, keeps components thin.
2. **Pre-session config mutation** — add `updateSessionConfig(patch)` (merge + `idle→configured`) vs granular `setMode`/`setDirection`/… vs local-form-state-until-Start. Default vote: **`updateSessionConfig(patch)`** — the store is the live source of truth (ARCH-007 holds mode/direction/models top-level), so ModeToggle + selectors write to it; one merging action beats 4 setters.
3. **Cascade-enabled criterion** — all three stages (`stt && translation && tts`) configured, or any? Default vote: **all three** — cascade is the full STT→Translation→TTS pipeline; a missing stage can't run a turn.
4. **Config source** — read `providerHealth` from the store (D.1 App bootstrap already populated it) vs SessionSetup re-fetches. Default vote: **read from store, no re-fetch** — single fetch point (App), components render from state (clean separation); config-retry-on-failure is a later/G concern.
5. **Start in-flight status** — use the `'starting'` `sessionStatus` during the create round-trip (disable Start while starting; revert to `'configured'` on error) vs a local pending flag. Default vote: **use `'starting'`** — the enum exists for exactly this; keeps the single-source-of-truth in the store.
6. **Direction options** — en→es + es→en only (no same-language). Default vote: **two options** (ARCH-002 EN↔ES).
7. **Start/End orchestration — extract to a testable `state/sessionActions.ts` vs inline in `SessionSetup`?** The Start/End flows carry real logic (clearErrors → starting → createSession → sessionStarted / addError+revert; end → sessionEnded + warning surfacing) that deserves a deterministic unit test without a render. Default vote: **extract `startSession(deps, formInput)` + `endSession(deps)` into `state/sessionActions.ts`** (mock store + mock `sessionsApi`); `SessionSetup` becomes a thin caller. Keeps the component manual-smoke (render test still D.7) while the orchestration is TDD'd. (If you'd rather keep it inline + lean on D.7's render test, say so — but then the stale-errors/revert behavior isn't unit-pinned.)
