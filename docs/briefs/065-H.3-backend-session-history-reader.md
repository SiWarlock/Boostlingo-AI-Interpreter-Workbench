# /tdd brief — session_history_reader (H.3-backend; persisted-session list + disk reader, Option B)

> **Backend slice (`server/`).** H.3 = session-history timeline (a past-sessions view). This is the backend half:
> a `SessionPersistenceReader` (Option B — read persisted sessions from DISK) + a `GET /api/sessions` list endpoint.
> **⭐ SAFETY-TOUCHING:** persistence is **write-only today** — this is the FIRST request-reachable DISK-ENUMERATION +
> DISK-DESERIALIZATION of persisted sessions (a new ARCH-019 boundary). It gets the mandatory `security-reviewer` pass
> and **its own commit** (don't bundle).

## Feature
A `SessionPersistenceReader` that enumerates `SESSION_DATA_DIR` for `session_*.json`, deserializes each (degrade-safe),
and a `GET /api/sessions` endpoint returning a lightweight summary list of persisted sessions (most-recent first) — so
the H.3-frontend can list past sessions (even across a server restart, when the in-memory store is empty).

## ⭐ Pre-orient at Step 1 (the persistence map — confirm)
- **Write side** (`Sessions/SessionPersistenceWriter.cs`): filename `session_<StartedAt UTC yyyyMMddTHHmmssZ>_<short-id>.json`
  in `SESSION_DATA_DIR` (default `../../data/sessions`, direct `builder.Configuration["SESSION_DATA_DIR"]`, singleton-wired
  at `Program.cs`). Serializes the full `InterpretationSession` via `Common/JsonDefaults`. **Two-layer path guard** (§11):
  `IsValidSessionId` ASCII allowlist `^[A-Za-z0-9_-]+$` pre-FS + canonicalized separator-terminated `StartsWith` under root.
- **Read side TODAY:** NONE. `GET /api/sessions/{id}` reads the **in-memory** `SessionStore` only (`SessionStore.Get` →
  `ConcurrentDictionary`); zero disk-read exists. `SessionStore.ActiveSessionIds()` lists in-memory ids only.
- **Reusable degrade pattern:** `Cost/PricingLoader.Load` + `Evaluation/EvaluationPhraseStore.Load` — pre-read size guard
  + filtered catch (`IOException|JsonException|UnauthorizedAccessException|ArgumentException|NotSupportedException|SecurityException`,
  NEVER bare) + `Result<T>` return + `JsonDefaults` deserialize. Confirm these at Step 1.

## Design (the honest, degrade-safe disk reader)
- **`SessionPersistenceReader(string dataDir)`** (singleton-wired like the writer): 
  - `Result<IReadOnlyList<InterpretationSession>> ReadAll()` (or summaries — see Q1) — enumerate `dataDir` for
    `session_*.json`, deserialize each via `JsonDefaults`. **Degrade PER-FILE:** a corrupt/oversize/unreadable file is
    SKIPPED (logged server-side, single-lined), not a whole-list failure — one bad file can't blank the history (Q3).
    A missing dir → empty list (not an error). Wrap enumeration + each read in the §3 filtered catch.
  - **Per-file size guard** (pre-read, like §3) — reject a file over a cap before deserializing (DoS guard; sessions can
    be larger than pricing.json → confirm the cap, Q5).
  - **Path safety:** enumerate `dataDir` ONLY (no recursion, no following symlinks beyond the canonicalized root); each
    enumerated path canonicalized + `StartsWith(dataDir)` checked (reuse the §11 guard) — defense even though enumeration
    is dir-scoped. A filename not matching `session_*.json` is skipped.
  - Optionally `Result<InterpretationSession?> ReadById(string id)` (the by-id disk read) — but the GET /{id} disk-fallback
    wiring is Q2 (likely a follow-up; this slice is the LIST).
- **`GET /api/sessions`** (`SessionsController`, thin per ARCH-008): calls the reader (via a service), maps `Result`→DTO at
  the boundary (§16 — never serialize a `Result`; success→`Ok(list)`, failure→sanitized `UiError`), returns the summary
  list (Q1 shape) ordered by `StartedAt` desc (Q4).
- **Invariant #3 (no audio/secret) holds structurally** — the reader deserializes the same `InterpretationSession` the
  writer wrote (no audio/secret fields exist on it); **keep the `SessionPersistenceTests` sentinel scan current** and add a
  read-side sentinel assertion (deserialize-then-serialize round-trip stays clean). The reader is NOT a sanitization
  boundary (it reads already-clean data) but it IS a path/DoS boundary.

## Acceptance (RED-first; deterministic — against a temp dir of fixture JSON, never a live FS dependency)
- [ ] **`ReadAll` enumerates + deserializes valid session files** → the persisted `InterpretationSession`s (round-trip
      fidelity via `JsonDefaults`). RED via a temp dir with 2-3 fixture `session_*.json`.
- [ ] **Degrade per-file:** a corrupt/non-JSON file + an oversize file in the dir are SKIPPED; the valid ones still return.
      A missing dir → empty list (not a throw). Pin both.
- [ ] **Path/pattern safety:** a non-`session_*.json` file is skipped; a path-escaping name (if constructable) is rejected
      (reuse §11). The reader never reads outside `SESSION_DATA_DIR`. Pin via a sentinel file outside the dir staying unread.
- [ ] **`GET /api/sessions` returns the summary list** (Q1 shape) ordered by `StartedAt` desc; `Result`→DTO mapped at the
      boundary (success→`Ok`; a reader failure→sanitized `UiError`, no path/exception leak). Verify the wire with
      `WebApplicationFactory`.
- [ ] **Empty dir / no sessions → `200` with an empty list** (not 404/500).
- [ ] **Sentinel maintained:** a read-side scan proves no `sk-`/`ek_`/audio in the response (round-trip clean); the
      `BuildFullSession` fixture stays current with the model.

## Things to flag at Step 2.5
1. **⭐ Response shape — Option B lightweight `SessionListItem` (my default) vs Option A full `InterpretationSession[]`.**
   My vote: **B** — a list endpoint should return a SUMMARY (`{sessionId, label, startedAt, endedAt?, turnCount, modes}`),
   not every session's full turns+latency events (payload hygiene; the H.3-frontend lists summaries + drills into detail via
   the existing `GET /{id}`). A is simpler (no new DTO, reuse the model) but ships wasteful full payloads. Confirm.
2. **GET /{id} disk-read fallback — this slice or a follow-up?** Today `GET /{id}` reads in-memory only → a past session
   (post-restart) 404s. The H.3-frontend drill-in needs a disk-fallback. My vote: **follow-up** (keep this slice the
   list + the reader; expose `ReadById` on the reader but wire the GET /{id} fallback in a separate small slice when
   H.3-frontend needs it). Confirm vs including it now.
3. **Degrade granularity — skip-per-file (my default) vs fail-whole-list.** My vote: **skip-per-file** (one corrupt file
   can't blank the history; log it server-side single-lined). Confirm.
4. **List ordering — `StartedAt` desc (most-recent-first, my default).** Confirm.
5. **Per-file size cap** — sessions can exceed pricing.json's 1 MB (many turns + latency events). My vote: a dedicated
   `SESSION_MAX_READ_BYTES` (e.g. 8 MB) cap. Confirm the value + whether it's env-configurable.
