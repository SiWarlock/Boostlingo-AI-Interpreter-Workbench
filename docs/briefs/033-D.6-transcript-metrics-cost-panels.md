## THE LOAD-BEARING DESIGN — metrics sourcing (read before Step 2.5)
Grounded in the backend `MetricsAggregator` (`server/.../Metrics/MetricsAggregator.cs`) + `ModeSummary` (`SessionModels.cs`). **Three sources, do NOT conflate:**

1. **Per-stage cascade latency (LIVE) → from the store.** The cascade WS streams `latency` events; `appendLatencyEvent` already stored them in `currentTurn.latency.stages[name]` (server-computed `relativeMs`). The MetricsPanel renders these directly — **do NOT recompute** (lesson §7: `relativeMs` is per-event display; the backend owns the math).

2. **Top-level client-timing deltas (LIVE) → frontend-computed, because the backend CANNOT for cascade.** `speechEnd→playback`, `speechEnd→firstAudio`, `totalTurn` need the **client-side** events `turn.recording.started`/`turn.recording.stopped`/`playback.started`. For cascade, those NEVER reach the backend (no client→server latency channel — the cascade turn is persisted by the WS on `done`), so the backend's `ModeSummary.AvgSpeechEnd*` is **`n/a` for cascade**. The frontend HAS these events (it stamps recording.started/stopped + playback.started, D.5) → it computes the deltas itself. This is NOT duplicating the aggregator (the backend structurally can't do it for cascade); it's the **only** source. Use the SAME `Between(from,to) = to.timestamp − from.timestamp` (absolute-timestamp subtraction, **never `relativeMs`**); `speechEnd→playback` is browser-clock-only (clean); `speechEnd→firstAudio` mixes browser-stopped + server-firstAudio (cross-clock — for the localhost demo the clocks match; per ARCH-013 disclose skew, never clamp negatives). A metric whose endpoint event is absent → `null` → `n/a`.

3. **Session averages → from the backend `GET /api/sessions/{id}/summary` (canonical).** Add `sessionsApi.getSummary(id) → SessionSummary`. The MetricsPanel's "session averages by mode" reads `ModeSummary` (per-stage avgs ARE present for cascade; `AvgSpeechEnd*` is `n/a` for cascade per #2). This is where the backend canonical aggregation lives — read it, don't recompute.

**Net:** per-stage = store (live); top-level = frontend-compute (live, the cascade headline); session-avgs = backend summary. The realtime path (E) reports client events via `POST …/events`, so the backend computes realtime top-level canonically — but that's E; D.6 is cascade-live.
