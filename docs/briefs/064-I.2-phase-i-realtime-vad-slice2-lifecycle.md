# /tdd brief — realtime_server_vad_lifecycle (I.2 slice 2; Phase-I realtime, the multi-segment auto-turn lifecycle)

> **Frontend slice (`web/`).** Slice 2 of 2 for Phase-I realtime auto-VAD. Slice 1 (`3ff6bed`) shipped the
> toggle + `turn_detection: server_vad` config + the single-utterance `settled` latch (one turn per Start). **This
> slice wires the MULTI-SEGMENT auto-turn lifecycle**: under server-VAD the server auto-detects EACH speech segment
> (`speech_started`/`speech_stopped`/`committed` → auto `response.created` → `response.done`), so ONE recording
> session yields MULTIPLE turns. Remove the single-utterance bound; begin a turn per server-detected segment. This is
> the project's most intricate slice — read the verification + design flags carefully.

## Feature
In auto-VAD mode, map each server-detected speech segment to its own `InterpretationTurn`: begin a turn on
speech-start, finalize on that segment's `response.done` (+ the §26 `/complete` cost POST + `/events` per auto-turn),
and be ready for the next segment — replacing slice-1's single-utterance `settled` latch + its DEV-warn drop.

## Design intent (confirm/refine at Step 2.5 after the Step-1 verification)
- **Remove the slice-1 single-utterance `settled` latch + its DEV warn** (they were the interim bound). The controller
  now processes ALL segments in an auto-VAD recording.
- **Begin a turn per server-detected segment:** on `speech_started` (the new-segment marker), `store.beginTurn(...)`
  (a fresh per-turn sink, §16) — so each segment is its own `InterpretationTurn`. Finalize on that segment's
  `response.done` (the sink's existing `completeTurn` + the controller's §26 `/complete` cost POST + `/events`).
  Multiple segments → `turns[]` grows by one per segment. (Confirm the `speech_started`-before-`response.created`
  ordering pins the begin correctly — a turn must exist before its transcript deltas arrive.)
- **Correlate segments if needed:** if deltas/`response.done` carry a `response_id`, use it to route events to the
  right turn (guards against interleave); if strictly sequential (one segment fully completes before the next starts),
  a simple current-segment model suffices. Confirm from the Step-1 sequence.
- **The auto-mode metrics speech-end anchor** (slice-1 carry-forward): under server-VAD the user's manual Stop is NOT
  the speech-end — the server's `speech_stopped`/`committed` is. Anchor the realtime responsiveness metric on the
  server signal (not `turn.recording.stopped`), mirroring the cascade §25 `stt.final` re-anchor. Flag at Step-2.5.
- **Stop in auto mode** (slice-1 deferred the guarded early-end): now that `committed` is observable, a manual Stop
  CAN force-close the current segment IFF no auto-commit seen (the guarded early-end). Optional — flag whether to
  include it or keep Stop a no-op.
- **Invariant #3 preserved:** audio via the media track; sink store-only; no audio bytes. `ek_` transient (#2).
