# /tdd brief — openai_translation_provider

## Feature
Implement the real `OpenAiTranslationProvider` (`ITranslationProvider`) behind the unchanged B.1 interface: a **streamed** OpenAI Responses API call (`POST /v1/responses`, `stream=true`) that emits `TranslationStarted → TranslationPartial* → TranslationFinal` from the SSE event stream, extracts usage tokens off `response.completed`, records the model used, and maps HTTP/exception failures to `ProviderError` via the existing `ProviderErrorMapper`. Second real provider of Phase C; reuses the C.1 shell+mapper pattern (lesson §18) but over **raw `HttpClient`** (not a callback SDK).

## Use case + traceability
- **Task ID:** C.2
- **Architecture sections it implements:** `ARCH-011` (§8 Translation path — streamed Responses API), `ARCH-012` (§9 `ITranslationProvider` contract, exception→`ProviderError`, `OpenAiTranslationOptions`).
- **Related context:**
  - **Reuses the C.1 pattern (lesson §18):** thin transport shell + pure unit-TDD'd mapper; reuse `ProviderErrorMapper.Map(ex, "openai", "translation")` (never re-map); match the `FakeTranslationProvider` (B.2) event contract (`TranslationStarted → TranslationPartial(TextDelta) → TranslationFinal(Text, InputTokens?, OutputTokens?)`).
  - **KEY DIFFERENCE vs C.1 — raw `HttpClient`, so the Q2 fork lands on (a).** OpenAI has no injection-blocking SDK; inject a **mock `HttpMessageHandler`** returning a **canned SSE byte stream** and TDD the streaming parse end-to-end (first-token timing, multi-delta aggregation, usage extraction, error statuses). This is higher-fidelity than C.1's pure-parse-only fallback, at the same cost. The actual network call stays the thin shell; everything else is tested.
  - **Status-bearing errors (no Q5 gap here):** raw `HttpClient` surfaces `HttpRequestException` with `StatusCode` (or the mock handler returns a real status), so `ProviderErrorMapper` maps `429→translation.rate_limited`(retryable) / `401/403→translation.auth` / `400→translation.invalid_request` correctly — unlike the Deepgram C.1 Q5 degrade. (Carry-forward "verify provider SDK surfaces" item, origin C.1.)
  - **TDD posture:** real *network* is exempt/manual-smoke, but with the mock handler the SSE parse + mapping + request-body construction are all deterministic → TDD them.

## Acceptance criteria (what "done" means)
- [ ] `OpenAiTranslationProvider : ITranslationProvider` exists; `TranslateAsync` streams the Responses API and yields `TranslationStarted` then ≥1 `TranslationPartial` then one `TranslationFinal` — **genuinely streaming, never buffered-then-emitted** (forbidden patterns #3/#4; ARCH-011 "the translation stage must stream").
- [ ] **SSE mapping (pure, unit-tested via canned stream):** `response.created → TranslationStarted`; **first** `response.output_text.delta → TranslationPartial` (this is the `translation.first_token` moment — orchestrator stamps it, B.4); subsequent deltas → `TranslationPartial(delta)`; `response.completed → TranslationFinal(aggregatedText, inputTokens, outputTokens)`.
- [ ] **Usage extraction:** `TranslationFinal.InputTokens`/`OutputTokens` are read off the `response.completed` usage block; absent usage → `null` tokens (cost estimator degrades, per B.5 — do NOT fabricate).
- [ ] **Final text = aggregation** of the streamed deltas (the contract is `TranslationFinal.Text` carries the full translation), not a separate re-fetch.
- [ ] **Request body** sets `model` (from request/options), `stream=true`, **nested** `reasoning: {effort: "minimal"}` + `text: {verbosity: "low"}` (Step-1 verified — top-level `reasoning_effort` is Chat-Completions-only + rejected by `/v1/responses`), and a **faithful-interpreter system instruction** (output ONLY the translation, no framing) + the source text as input; targets the SourceLanguage→TargetLanguage direction.
- [ ] **Model recorded:** the model used is available to the caller for `TranslationModelUsed` (ARCH-011) — confirm via the request/event (orchestrator records per turn).
- [ ] **Error mapping:** HTTP/exception → `TranslationFailed(ProviderErrorMapper.Map(ex, "openai", "translation"))`; `429→translation.rate_limited`(retryable), `401/403→translation.auth`(non-retryable). No new mapper codes.
- [ ] **No leak:** the OpenAI API key flows only from `OpenAiTranslationOptions` (server-side); never into any `TranslationFailed`/`ProviderError`/log line.
- [ ] All unit tests in `AiInterpreter.Tests/OpenAiTranslationProviderTests.cs` pass; full suite stays green (will be **171 + new**).
- [ ] `/preflight` clean.

## RED test outline (Step 2 — `AiInterpreter.Tests/OpenAiTranslationProviderTests.cs`)
Drive via a mock `HttpMessageHandler` returning a canned SSE stream (or a pure SSE-parse fn over synthetic chunks — Step 2.5 Q2).

1. **`streams_started_partials_final_in_order`** — a canned SSE stream (`response.created`, two `response.output_text.delta` "Hola"/" mundo", `response.completed`) → ordered `[TranslationStarted, TranslationPartial("Hola"), TranslationPartial(" mundo"), TranslationFinal("Hola mundo", …)]`. Why: ARCH-011 streaming pipeline; proves genuine streaming (multiple partials before final).
2. **`first_delta_is_first_partial`** — the first `output_text.delta` produces the first `TranslationPartial` (the orchestrator's `translation.first_token` source). Why: ARCH-011/013 first-token latency; forbidden #3 (real first arrival, not synthesized).
3. **`final_aggregates_deltas`** — `TranslationFinal.Text` == concatenation of all deltas. Why: the `TranslationFinal` contract (full translation).
4. **`usage_tokens_extracted_from_completed`** — `response.completed` usage → `TranslationFinal.InputTokens`/`OutputTokens` populated. Why: ARCH-014 cost (B.5 consumes these).
5. **`missing_usage_yields_null_tokens`** — `response.completed` without usage → `null` tokens (no fabrication). Why: B.5 degrade-don't-fabricate.
6. **`rate_limited_yields_failed_retryable`** — mock returns 429 → `TranslationFailed{Code="translation.rate_limited", Retryable=true, Stage="translation", Provider="openai"}`. Why: ARCH-012 table via mapper (status-bearing — no Q5 gap).
7. **`auth_error_yields_failed_nonretryable`** — mock returns 401/403 → `TranslationFailed{Code="translation.auth", Retryable=false}`. Why: ARCH-012 table.
8. **`request_body_sets_streaming_and_interpreter_prompt`** — assert the outgoing request JSON has `stream=true`, **nested** `reasoning.effort=="minimal"` + `text.verbosity=="low"`, the chosen `model`, the `input` source text, and a faithful-interpreter system instruction (contains "ONLY" + the languages). Why: ARCH-011 (latency-protecting params + output-only-translation); a behavioral pin (a missing `stream=true` would silently break streaming). _(Step-1 verified — top-level `reasoning_effort` is Chat-Completions-only + rejected by `/v1/responses`.)_
9. **`delta_split_across_reads_reassembles`** (Q3) — the same SSE delivered **1 byte per read** → identical ordered events. Why: proves `StreamReader` line reassembly across reads (no hand-rolled buffering that corrupts a split delta).
10. **`mid_stream_error_event_yields_failed`** (ADD, orchestrator Step-2.5) — `created`, one `delta`, then an in-band SSE **`error` event** → terminal `TranslationFailed` (no `TranslationFinal`). Why: error fidelity (ARCH-018) — a mid-stream streaming error is a real failure mode distinct from an HTTP status. Map to `translation.unknown` unless the error payload carries a mappable type/code (impl discretion; don't over-build).

10 tests — lean, but cover streaming-honesty + usage + errors (HTTP + in-band) + request shape. **API-shape (Step-1 verified):** no `[DONE]` sentinel on `/v1/responses` (stream ends after `response.completed`); usage at `response.usage.{input_tokens,output_tokens}` (snake_case). Do NOT re-test the mapper truth table (B.1) or `FakeTranslation` per-variant behavior (B.2) — lesson §17.

## Things to flag at Step 2.5
Defaults pre-loaded. **Q1/Q2 are the genuine forks.**

1. **HttpClient construction seam.** (a) `IHttpClientFactory` typed/named client registered in `Program.cs` (idiomatic, testable via injected handler); (b) a constructor-injected `HttpClient` the test supplies directly (no `Program.cs` touch this slice). **Default vote: (b) constructor-injected `HttpClient`** — keeps C.2 out of `Program.cs` (consistent with the "DI wiring is C.4" boundary) and tests inject a mock-handler-backed `HttpClient` directly. C.4 decides the factory registration when it wires real-vs-fake. (If you prefer (a) now, it's fine — just the one HttpClient DI line, NOT the provider swap.)
2. **SSE parse seam placement.** `internal static OpenAiTranslationMapping` (pure SSE-line/event → `TranslationEvent`, reusing C.1's `InternalsVisibleTo`) vs inline private parsing tested only through `TranslateAsync` (mock handler). **Default vote: a pure `internal static` parse helper** (matches lesson §18) — but since the mock handler already drives `TranslateAsync` end-to-end, a thin private parser tested via the public method is acceptable if cleaner. Your call on the split; keep the SSE-framing logic unit-reachable.
3. **SSE framing robustness.** Responses API SSE = `event:`/`data:` lines, `[DONE]` sentinel, possibly multi-line `data:` + keep-alive blanks. **Default vote: parse line-buffered `data:` JSON, ignore non-JSON/keep-alive lines, stop on stream end/`[DONE]`** — pin one "delta split across reads / blank keep-alive line" case so partial-chunk reassembly is covered. Don't over-engineer beyond what the canned stream exercises.
4. **Direction → prompt.** The faithful-interpreter system instruction must encode SourceLanguage→TargetLanguage (EN↔ES). **Default vote: a single parameterized instruction** ("Translate the user message from {source} to {target}; output ONLY the translation") — confirm it outputs no framing. (LLM output *quality* is exempt from unit assertion — pin the request shape, not the translation correctness.)
5. **`response.completed` vs `response.output_text.done` for final.** Confirm the terminal event carrying the aggregated text + usage. **Default vote: aggregate deltas for `Text` + read usage off `response.completed`** (don't depend on a separate full-text event that may not arrive). Verify against the actual API at Step 1.
