# /tdd brief — env_autoloader (G.2b; backend dev-ergonomics)

> **Small, independent backend slice** (host-wiring; no cross-area, no safety-invariant change). Keeps the idle backend impl productive while H.3's A/B is with the user. Lands atomically before any H.3-backend.

## Feature
Auto-load the repo-root `.env` at backend startup so a clean clone runs with plain `dotnet run` — removing today's manual `set -a && source ../../.env && set +a && dotnet run` friction (the live-server start ritual). Discharges the G.2b carry-forward.

## Context / root cause
There is **no `.env` auto-loader** today (decided deferred at A.5). A clean clone (or the lead's live server) must source `.env` by hand before `dotnet run`, or every provider key / model env var is absent → config-gating disables both modes. The `Program.cs` env→section bridge (server lesson §4) already maps flat ARCH-028 env vars → Options sections — it just needs the env vars *present* in the process before it reads them.

## Use case + traceability
- **Task ID:** G.2b (`.env` auto-loader). **Architecture:** `ARCH-028` (config & secrets / `.env` contract), `ARCH-029` (build & run). No new contract surface; no model change.
- **Invariant #1 (keys server-side only):** `.env` loads into the **process environment** (backend), never a response body or log — unchanged trust boundary. `.env` stays gitignored (only `.env.example` is tracked).

## Acceptance (RED-first where deterministic; host-wiring per lesson §4 + degrade §3)
- [ ] At startup, **before** the config/env-bridge reads env vars, the host loads `.env` from the repo root if present → the ARCH-028 vars are in `Environment` → the existing §4 bridge binds the Options (a clean `dotnet run` works with NO manual sourcing). Pin the deterministic seam: point the loader at a `.env` **fixture** → assert the expected vars are populated (and flow through to a bound Options value).
- [ ] **Absent `.env` → no crash** (degrade-don't-crash, §3) — production/CI has no `.env` and uses real process env vars; startup proceeds normally.
- [ ] **Already-set process env vars are NOT overridden** by `.env` (Step-2.5 Q1) — an explicit `export` (the lead's `OPENAI_TRANSLATION_MODEL` override) + CI/prod env must win. Pin precedence with a set-before-load test (lesson §4 — set/unset serialized).
- [ ] No secret leak: keys stay process-side (a sentinel-style check that the loader path doesn't echo/log key values).
- [ ] `/preflight` (backend) clean; full `dotnet test` green.

## Things to flag at Step 2.5
1. **Override vs fill-gaps precedence.** My vote: **fill-gaps only — do NOT override already-set process env vars** (explicit `export` + CI/prod must win; `.env` is the fallback). Confirm + pin.
2. **Path resolution.** `.env` is at repo root; `dotnet run` cwd varies (`server/AiInterpreter.Api` vs repo root via `--project`). My vote: a **walk-up-from-content-root** resolution (robust to cwd) rather than a fixed `../../.env`. Confirm the exact mechanism.
3. **Library vs hand-rolled.** A small loader lib (e.g. **DotNetEnv** — standard, tiny; **verify its current API + the no-override option via Context7** before wiring) vs a ~20-line hand-rolled parser (no dep, but reinvents quoting/escaping). My vote: the lib if its no-override + path options are clean; else hand-roll. Confirm.
4. **Dev-only vs always-load.** My vote: **always load-if-present** (simplest; harmless in prod where no `.env` exists) rather than gating on `ASPNETCORE_ENVIRONMENT==Development`. Confirm.
