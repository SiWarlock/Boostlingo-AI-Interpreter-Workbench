# /tdd brief — session_lifecycle_endpoints (B.9c-i)

## Feature
`SessionsController` **session-lifecycle** routes + a `SessionService` that orchestrates them over the existing seams: `POST /api/sessions` (create), `GET /api/sessions/{id}` (fetch), `POST /api/sessions/{id}/end` (finalize summary + persist MUST), `GET /api/sessions/{id}/summary` (recompute on demand). Establishes the **`Result`/`Result<T>`→HTTP-DTO mapping** (never serialize a `Result`) and adds the deferred B.8 `Result<T>` `SanitizeResult` overload (this is its first consumer). Not a safety slice (the path-guard/sentinel safety lives in B.7a; the sanitization boundary in B.8/B.9a).

> **B.9c is split into two** (the controller is large; session-CRUD vs turn-lifecycle are separable sub-resources): **B.9c-i** (this — session lifecycle) and **B.9c-ii** (turn lifecycle: `POST …/turns`, `…/complete`, `…/events` + backend-owned turnId + events collection-size bounds). B.9 overall: a (handler) / b (config) / c-i (sessions) / c-ii (turns).

1. **`create_session_returns_session_with_assembled_profile`** — POST a request → 200/201; response has a server `sessionId` (`^[A-Za-z0-9_-]+$`), `startedAt`, and a `config.providerProfile` filled from request models + Options defaults (sttProvider=deepgram, ttsVoice=alloy, etc.); enums serialize camelCase (`"realtime"`).

6. **`end_persistence_failure_degrades_to_uierror`** — force a write failure (unwritable `SESSION_DATA_DIR`) → response is a `UiError` (`persistence.failed`, no path/`Result.Error` leak); the in-memory end still happened.

## Things to flag at Step 2.5
1. **Create/Get response shape.** My default vote: **return the full `InterpretationSession`** (camelCase) from both create + get — it's the ARCH-005-record-as-API-JSON principle (ARCH-005 line 209); the ARCH-009 create example `{sessionId, startedAt, config}` is an abbreviation (those fields are a subset of the full record). Alternative: a focused `CreateSessionResponse`. Default: full record.
2. **`Result`→DTO mapping placement + the `Result<T>` overload.** My default vote: a `ResultExtensions` helper (`ToActionResult(this Result<T>, ErrorSanitizer, okSelector)` / `Result` variant) that maps success→`Ok(dto)` and failure→the sanitized `UiError` at the right status; add the `Result<T>` `SanitizeResult` overload to `ErrorSanitizer` now (first consumer = `/end`). Flag if you'd rather put the mapping on a controller base class.
3. **`/end` summary snapshot.** My default vote: `SessionService.End` = `store.End(id)` (stamps `EndedAt`) → `summaryService.Compute(ended)` → snapshot via a focused `SessionStore.SetSummary(id, summary)` (record-swap under the gate, mirrors `End`) so `GET /{id}` after end reflects it → `writer.WriteAsync(finalSession)` (MUST). Flag if you'd rather persist the summary-bearing session without re-storing it (GET /summary recomputes anyway).
4. **`/end` persistence-failure semantics.** My default vote: the in-memory end (EndedAt + summary) **always** happens; a `WriteAsync` failure surfaces as a `UiError` (`persistence.failed`) in the response (Flow F shows save success/failure) but is **not** a 500 — the session ended, the save warned. Flag if you'd rather 200-with-warning vs an error status.
5. **Unknown-id error code.** My default vote: a **new `session.not_found`** code (fits `<stage>.<reason>`; I add it to ARCH-018) → 404 `UiError`. This is a controller-returned 404 (routed path, missing resource) — distinct from the framework 404/405 for *unrouted* paths (those stay ProblemDetails per the B.9a decision). Flag if you'd rather reuse a generic code.
6. **`ProviderProfile` assembly location.** My default vote: in `SessionService` (reads the A.2 `IOptions` + request models → `SessionConfig`), keeping the controller thin. Confirm.
