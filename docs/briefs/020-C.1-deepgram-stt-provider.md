# /tdd brief — deepgram_stt_provider

## Feature
Implement the real `DeepgramSttProvider` (`ISttProvider`) behind the unchanged B.1 interface: a **live-WebSocket** streaming path (`nova-3`, `language=multi`, interim + final segments) and a **pre-recorded REST fallback** path (single final, no interim), with SDK/HTTP exceptions mapped to `ProviderError` via the existing `ProviderErrorMapper`. First real-provider slice of Phase C — it sets the real-provider TDD pattern reused by C.2/C.3.

## Use case + traceability
- **Task ID:** C.1
- **Architecture sections it implements:** `ARCH-011` (§8 STT path — live WS + fallback), `ARCH-012` (§9 `ISttProvider` contract, exception→`ProviderError` mapping, `DeepgramOptions`), `ARCH-030` (§4 capture — fallback container auto-detect, no transcoding).
- **Related context:**
  - Plugs into the **shipped + proven** B.1 seam: `ISttProvider`/`SttEvent` (`Providers/Abstractions/`), `ProviderErrorMapper` (single owner of the ARCH-012 table; `SafeMessage` never echoes `ex.Message`), and the B.10 `ProviderBoundaryTests` interface-level suite that **C.5 extends** with real-provider conformance.
  - **Match the fake contract** (`FakeSttProvider`, B.2): `SttStarted` → `SttPartial*` → `SttFinal`; the empty-transcript case is `SttFinal("")` (the orchestrator owns `cascade.empty_transcript`, **not** the provider).
  - **TDD posture (settled, lead-confirmed):** real-provider *network calls* are exempt from unit TDD (manual smoke). TDD the **deterministic surfaces**: the pure Deepgram-result→`SttEvent` mapping, the pre-recorded REST response parsing, and the exception→`ProviderError` wiring. The live-WS transport shell (connect / subscribe / send frames / finish) is **manual-smoke only**. Keep it LEAN — no heavy HTTP-integration suite (build-timebox).

## Acceptance criteria (what "done" means)
- [ ] `DeepgramSttProvider : ISttProvider` exists; `TranscribeAsync` routes a **streaming linear16** request to the live-WS path and a **pre-recorded blob** request to the REST path (routing rule settled at Step 2.5).
- [ ] **Live mapping (pure, unit-tested):** a Deepgram interim result → `SttPartial(text)`; a final/utterance-end result → `SttFinal(text)`; an empty/whitespace final → `SttFinal("")` (never `SttFailed`, never `cascade.empty_transcript`).
- [ ] **Pre-recorded path:** emits ordered `SttStarted` then a single `SttFinal(transcript)` — **no `SttPartial`**; empty transcript → `SttFinal("")`.
- [ ] **Error mapping:** SDK/HTTP failures are caught and emitted as `SttFailed(ProviderErrorMapper.Map(ex, "deepgram", "stt"))` — `429`→`stt.rate_limited`(retryable), `401/403`→`stt.auth`(non-retryable). No new mapper codes unless the SDK forces it (flag at Step 9 if so → cross-doc on ARCH-012).
- [ ] **No leak:** the Deepgram API key flows only from `DeepgramOptions` (server-side); it never appears in any `SttFailed`/`ProviderError`/log line (reusing `ProviderErrorMapper` gives this by construction).
- [ ] Live-WS shell builds the Deepgram schema from the request + options (`model=nova-3`, `language=multi`, `smart_format=true`, `interim_results=true`, `encoding=linear16`, `sample_rate=<request>`, `channels=1`, `utterance_end_ms`).
- [ ] All unit tests in `AiInterpreter.Tests/DeepgramSttProviderTests.cs` pass; full suite stays green (currently **161**).
- [ ] `/preflight` clean (`dotnet format --verify-no-changes && dotnet build && dotnet test`).
- [ ] NuGet `Deepgram` v6.6.x added to `AiInterpreter.Api.csproj` (only Swashbuckle/OpenApi present today).

## RED test outline (Step 2 — `AiInterpreter.Tests/DeepgramSttProviderTests.cs`)
Pin the **contract**, not the SDK internals. Mechanism (mock `HttpMessageHandler` vs pure-parse fn) is Step-2.5 Q2 — keep whichever is leaner against v6.6.x.

**Group 1 — Deepgram result → `SttEvent` mapping (pure, synthetic inputs):**
1. **`interim_result_maps_to_partial`** — interim (is_final=false) "hel" → `SttPartial("hel")`. Why: ARCH-011 STT path (partial per interim).
2. **`final_result_maps_to_final`** — final/utterance-end "hello world" → `SttFinal("hello world")`. Why: ARCH-011 (final per segment).
3. **`empty_final_maps_to_final_empty`** — final with empty/whitespace transcript → `SttFinal("")` (NOT `SttFailed`; NOT `cascade.empty_transcript`). Why: matches `FakeStt.EmptyFinal` + `ProviderErrorMapper.EmptyTranscript` is orchestrator-owned.

**Group 2 — Pre-recorded REST fallback:**
4. **`prerecorded_yields_started_then_single_final`** — canned Deepgram REST result (transcript "hola mundo") → ordered `SttStarted` then `SttFinal("hola mundo")`, **no `SttPartial`**. Why: ARCH-011 fallback (single final, no interim).
5. **`prerecorded_empty_yields_final_empty`** — canned REST result, empty transcript → `SttFinal("")`. Why: empty-contract consistency across paths.

**Group 3 — Exception/HTTP → `ProviderError` (catch-path wiring):**
6. **`rate_limited_yields_failed_retryable`** — a 429 → `SttFailed` with `Code="stt.rate_limited"`, `Retryable=true`, `Stage="stt"`, `Provider="deepgram"`. Why: ARCH-012 table via `ProviderErrorMapper`.
7. **`auth_error_yields_failed_nonretryable`** — a 401/403 → `SttFailed` `Code="stt.auth"`, `Retryable=false`. Why: ARCH-012 table; non-retryable (mirrors the B.10 non-retryable-403 pin).

7 tests — lean. If Step-2.5 Q2 lands on "pure-parse fn" (SDK won't inject `HttpClient`), Group 2/3 test the parse fn + the catch→mapper unit directly; the mapper truth table itself is already covered by B.1 — do **not** re-test it (lesson §17).

## Things to flag at Step 2.5
Defaults pre-loaded — take them (fast path) or ping back with evidence. **Q1/Q2 are genuine forks; Q3–Q6 are mostly confirm-only.**

1. **Live-WS vs pre-recorded-REST routing inside `TranscribeAsync`.** SttRequest is shared (ARCH-012: the fallback "wraps the blob as a single sequence"). Branch on the request: `Encoding=="linear16"` (streaming PCM) → live WS; a recorded-container `ContentType` (e.g. `audio/webm`, non-linear16) → REST single-shot. Alt: a dedicated flag/second method. **Default vote: branch on `Encoding`/`ContentType` within `TranscribeAsync`** — keeps one `ISttProvider` seam; the C.4 WS orchestrator and the C.5 blob orchestrator just supply different requests. (Confirm the exact discriminator field.)
2. **REST testable-seam mechanism.** (a) Inject a mock `HttpMessageHandler`/`HttpClient` into the Deepgram REST client if v6.6.x accepts one (tests the SDK's own parsing too); (b) extract a pure `ParsePrerecorded(responseDto) → IEnumerable<SttEvent>` and test that, leaving the REST call a thin shell. **Default vote: (a) if the SDK cleanly takes an injected `HttpClient`; else (b).** Don't fight the SDK — fall back to (b) to stay lean. (This choice sets the C.2/C.3 pattern.)
3. **Pure-mapping seam placement.** `internal static DeepgramSttMapping` + `InternalsVisibleTo("AiInterpreter.Tests")`, vs private methods tested only through `TranscribeAsync`. **Default vote: `internal static` helper + `InternalsVisibleTo`** — lets the pure mapping be unit-tested without a live socket (the strategy the lead approved). Confirm no `InternalsVisibleTo` already exists.
4. **Callback→`IAsyncEnumerable` bridge.** The Deepgram WS client is event/callback-based; `ISttProvider` is `IAsyncEnumerable<SttEvent>`. Bridge via `System.Threading.Channels.Channel<SttEvent>` (writer in callbacks, reader yields with `[EnumeratorCancellation]`). **Default vote: `Channel<SttEvent>`** — standard bridge; the bridge wiring is manual-smoke (live socket), but the mapper the callbacks call is unit-tested. Confirm cancellation completes the channel + finishes/closes the socket (no leak — ties to G.4 stability).
5. **SDK exception surface for the mapper.** Does v6.6.x surface HTTP errors as status-bearing `HttpRequestException` (mapper handles directly) or an SDK-specific exception? **Default vote: catch the SDK exception, extract the status if exposed, route through `ProviderErrorMapper.Map`; if no status is available, accept the mapper's `{stage}.unknown` fallback.** Do NOT expand the mapper table speculatively — only if a real, mappable status exists (and then flag the cross-doc change).
6. **Empty-transcript ownership (confirm-only).** Provider emits `SttFinal("")`; the orchestrator (B.4) owns the `cascade.empty_transcript` short-circuit. **Default vote: confirmed** — settled by `FakeStt.EmptyFinal` + `ProviderErrorMapper.EmptyTranscript` (stage `"cascade"`). Don't emit `cascade.empty_transcript` from the provider.
