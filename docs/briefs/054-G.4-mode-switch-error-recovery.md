# /tdd brief — mode_switch_error_recovery (G.4; user-reported live Finding)

> **Real-key-smoke bug-fix (frontend; HIGH — UX-breaking, "must refresh to recover").** A user-reported live Finding in the mode-toggle, in the same 2c domain as 050. **Frontend-only**, independent of 050-backend (testable against a mocked `sessionsApi`) → runs in **parallel** with the in-flight 050-backend slice. Root cause pinned by a read-only investigation (below) — straight-to-`/tdd`.

## Feature
Make the mode toggle **self-recovering** and correct after a session ends. Three fixes:
- **(A)** `switchMode` must **not POST** when there is no active session (`sessionStatus === 'ended'`, `'ending'`, or no `sessionId`) — store-only config write, like the existing pre-session branch.
- **(B)** `ModeToggle` must **clear prior errors before dispatching** the switch (clear-before-retry, like `startSession`) so a failed switch doesn't strand the UI / force a page refresh.
- **(C)** A genuine switch failure must map to an **actionable `errorCopy`** code, not the generic "Something went wrong. Please retry." fallback.

## User-reported symptoms (the Finding)
- **Repro 1:** complete a turn → switch mode → red banner "Something went wrong. Please retry." *(This is the **404 from the not-yet-existing `/mode` endpoint** — resolved by **050-backend**, which is in flight. Listed for context; this slice does NOT depend on it.)*
- **Repro 2:** END the session → switch mode → same error → **UI STUCK; must REFRESH to recover.** *(This is the frontend bug this slice fixes.)*
- Banner text "Something went wrong. Please retry." = the **generic fallback** `errorCopy` (no mapped/actionable code).

## Root cause (confirmed — read-only investigation, 2026-05-31)
1. **`switchMode` missing the ended-session guard.** `web/src/state/sessionActions.ts` `switchMode` (~L77–110) early-returns a store-only write only when `sessionId === null` (~L85). But `sessionEnded()` (`sessionStore.ts` ~L139) sets `sessionStatus:'ended'` **without clearing `sessionId`** → a post-end toggle finds `sessionId` non-null → POSTs `api.setMode(sessionId, target)` → backend 404 → error. **The endpoint existing (050-backend) does NOT fix this** — switching after end shouldn't POST at all.
2. **No error clearing on a toggle attempt.** `web/src/components/ModeToggle.tsx` `onClick` (~L59–67) calls `switchMode(...)` **without** a preceding `store.clearErrors()` (contrast `startSession`, `sessionActions.ts` ~L41, which clears first). The catch in `switchMode` (~L97–105) `addError`s, but nothing clears it on retry → the banner lingers and errors can stack → "must refresh." `canToggleMode(turnStatus)` doesn't consider `errors`, so the toggle stays enabled-but-broken.
3. **`errorCopy` has no mapping for the failure code.** `web/src/components/errorCopy.ts` `errorCopy()` (~L8–42) has no case for `session.mode_switch_failed` or `http.404` → both hit the generic fallback (~L42). On an `ApiError` (the 404) `switchMode` surfaces the api's `http.404` uiError; on a non-`ApiError` throw it synthesizes `session.mode_switch_failed` (`sessionActions.ts` ~L100/104). Either way → unhelpful generic copy.

## Lessons-logged candidates anticipated
- **Convention candidate** — "A status-gated mutating action (e.g. `switchMode`) must guard **all** non-active statuses, not just the pre-session (`null`) case — a post-`ended` action otherwise hits a dead endpoint. And a mid-session action's error must be **clearable by the retry path** (clear-before-retry, like `startSession`) or a failed action strands the UI (the user must refresh). Map the failure to an actionable `errorCopy`, never the generic fallback."
