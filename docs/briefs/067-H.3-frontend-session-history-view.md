## Feature
A session-history view (panel/route) that fetches `GET /api/sessions` and renders the persisted sessions as a
list of `SessionListItem` summaries (label, started/ended time, turn count, modes), most-recent-first — so the user
can see past evidence runs even across a server restart.

## Design intent (confirm/refine at Step 2.5)
- **Fetch on demand** (the history view's mount / a refresh action) via a DI'd `historyActions.loadHistory()` (§7) →
  `sessionsApi.listSessions()` → local display state (Q3 — transient like the eval panel's result, NOT necessarily
  `UiSessionState`); a fetch failure routes a sanitized `UiError` to the store sink (§2).
- **Render** a `SessionHistory`/`HistoryPanel` component (thin, render+dispatch, ARCH-007): the list of
  `SessionListItem`s — label (or id), started/ended, turn count, the mode chips (blue=realtime/violet=cascade per the
  H.1 identity colors); empty list → an explicit "no past sessions" note; a fetch error → the sanitized banner.
- **Drill-in (Q2):** clicking a session to view its FULL detail needs `GET /{id}` — which today reads the in-memory
  store only (a past-restart session 404s). So scope the drill-in: either (a) list-only this slice (drill-in is the
  GET /{id}-fallback follow-up), or (b) drill-into-in-memory-only with a clear "full detail after restart needs the
  reader fallback" note. My vote: **(a) list-only** — ship the list now; the drill-in pairs with the BE GET /{id}
  fallback. Confirm.
