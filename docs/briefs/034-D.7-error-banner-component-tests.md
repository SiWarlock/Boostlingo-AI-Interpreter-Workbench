## Things to flag at Step 2.5
1. **jsdom env — per-file `@vitest-environment jsdom` (default vote) vs global `test.environment:'jsdom'`.** Per-file keeps the 92 existing node-env tests (fetch/FormData/Blob as node globals) untouched — lower regression risk. A global switch risks changing fetch/Response behavior for the existing client/store tests. **Default: per-file.**
2. **`errorCopy` location** — its own `errorCopy.ts` (pure, cleanly testable) vs inline in ErrorBanner. Default: **own module** (testable + reusable).
3. **Mic-denied test scope** — render the whole `App` vs just the relevant components (RecordingControls + ErrorBanner). Default: **the minimal component set** (RecordingControls + ErrorBanner + the store), mocking `getUserMedia` — less brittle than a full-App render.
4. **Dismiss affordance** — does ErrorBanner offer per-error dismiss (calling a store action)? Default: **render-only for D.7** (the D.1 LOW about index-keys/per-item-dismiss is deferred); a `clearErrors` on new-session already exists. Keep it simple.
