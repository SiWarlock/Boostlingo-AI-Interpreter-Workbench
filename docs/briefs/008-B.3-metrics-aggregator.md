# /tdd brief — latency_model_and_metrics_aggregator (B.3)

## Feature
Build the deterministic metrics layer: `LatencyEventFactory` (stamps a `LatencyEvent` with `clockSource` + `relativeMs` from a documented origin) and `MetricsAggregator` (computes the ARCH-013 universal + cascade-stage + realtime metrics off a turn's `LatencyEvent` list, MUST tier first, nice tier → `n/a` never error). Plus the shared metric vocabulary (`LatencyEventNames` constants) and the per-turn computed output type.

## Acceptance criteria (what "done" means)
- [ ] `LatencyEventFactory.Create(name, stage, clockSource, timestamp, origin, metadata?)` returns a `LatencyEvent` with `RelativeMs = round((timestamp − origin).TotalMilliseconds)`, clamped ≥ 0, and the supplied `ClockSource` preserved.
- [ ] `LatencyEventFactory.Stamp(name, stage, clockSource, origin, metadata?)` uses the injected `IClock.UtcNow` as the event timestamp (deterministic under a fake clock), same `relativeMs` math.
- [ ] `MetricsAggregator.Compute(IReadOnlyList<LatencyEvent>)` returns a `TurnMetrics` with all universal + cascade-stage + realtime metrics; each metric is `null` (→ `n/a`) when either endpoint event is absent. **Never throws** on missing/empty input.
- [ ] Universal metrics computed per ARCH-013: `speech_end_to_first_audio_ms`, `speech_end_to_playback_ms`, `total_turn_ms`, `audio_duration_ms`.
- [ ] Cascade-stage metrics per ARCH-013: `stt_first_partial_ms`, `stt_final_ms`, `translation_first_token_ms`, `translation_final_ms`, `tts_first_audio_ms`, `tts_complete_ms`.
- [ ] Realtime metrics per ARCH-013: `realtime_connect_ms`, `realtime_first_audio_delta_ms`, `realtime_first_transcript_delta_ms`, `realtime_playback_ms`.
- [ ] **Cross-clock pairs** (e.g. browser `turn.recording.stopped` → server `tts.first_audio`) compute via **absolute `Timestamp` subtraction**; small skew is tolerated and not clamped/rejected (ARCH-013 "small skew acceptable and disclosed").
- [ ] `LatencyEventNames` constants exist for every event the aggregator keys on (single source of the metric vocabulary; no magic strings in the aggregator).
- [ ] `IClock` → `SystemClock`, `LatencyEventFactory`, `MetricsAggregator` registered in `Program.cs` DI (available; production consumer is B.4 — see Wiring).
- [ ] All unit tests in `AiInterpreter.Tests/MetricsAggregatorTests.cs` pass.
- [ ] `/preflight` clean (`dotnet format --verify-no-changes && dotnet build` WAE 0W/0E `&& dotnet test`).

## RED test outline (Step 2) — `AiInterpreter.Tests/MetricsAggregatorTests.cs`
THIN-IF-NEEDED tier (ARCH-020): cover the MUST pairs + cross-clock + nice→`n/a`; don't gold-plate.

**`LatencyEventFactory`:**
1. **`Create_StampsRelativeMsFromOrigin`** — given `timestamp` 912ms after `origin` → `RelativeMs == 912`; `ClockSource`/`Stage`/`Name`/`Metadata` preserved.
   - Why: ARCH-013 schema (`relativeMs` from documented origin).
2. **`Stamp_UsesInjectedClock`** — fake `IClock` returning a fixed instant → `Timestamp == clock.UtcNow`, `RelativeMs` = fixed − origin.
   - Why: deterministic server-side "stamp on arrival"; consumes `IClock`.
3. **`Create_EventBeforeOrigin_ClampsRelativeMsToZero`** — `timestamp < origin` → `RelativeMs == 0` (no negative single-event latency; see Step-2.5 Q3).
   - Why: same-clock origin is monotonic; guards a nonsense display value.

**`MetricsAggregator`:**
4. **`Compute_CascadeMustPairs_ComputesUniversalAndStageMetrics`** — a cascade turn with all MUST events (+ nice `stt.first_partial`/`translation.first_token`/`tts.complete`) → assert specific ms values for `total_turn_ms`, `audio_duration_ms`, `speech_end_to_first_audio_ms` (= `tts.first_audio` − `recording.stopped`), `stt_final_ms`, `translation_first_token_ms`, `tts_first_audio_ms`, etc.
   - Why: ARCH-013 universal + cascade-stage formulas.
5. **`Compute_CrossClockPair_SpeechEndToFirstAudio_UsesAbsoluteTimestamps`** — `turn.recording.stopped` (`ClockSource.Browser`) + `tts.first_audio` (`ClockSource.Server`) → metric = wall-clock `Timestamp` diff; with a small server-ahead skew the value is computed (not clamped, not thrown).
   - Why: ARCH-013 "aggregator must handle cross-clock pairs explicitly… small skew acceptable."
6. **`Compute_MissingNiceTierEvents_YieldNullNotError`** — omit `stt.first_partial`/`translation.first_token`/`tts.complete` → those three metrics `null`; MUST metrics still computed.
   - Why: ARCH-013 "NICE … else `n/a` — never an error."
7. **`Compute_RealtimeMustPairs_ComputesRealtimeMetrics`** — a realtime turn (`realtime.session.connected`, `recording.stopped`, `realtime.first_audio_delta`, `playback.started`, `turn.completed`) → realtime metrics computed; **cascade stage metrics `null`** (absence-driven).
   - Why: ARCH-013 realtime tier; mode-implicit n/a.
8. **`Compute_EmptyEventList_AllMetricsNull_NoThrow`** — `[]` → every `TurnMetrics` field `null`, no exception.
   - Why: robustness; never-error contract.

## Things to flag at Step 2.5
1. **Aggregator math source — absolute `Timestamp` vs `relativeMs` deltas?** Options: subtract event `Timestamp` (DateTimeOffset) values; or subtract stamped `relativeMs` values. My default vote: **absolute `Timestamp` subtraction.** Cross-clock pairs (browser `recording.stopped` → server `tts.first_audio`) are only meaningful as wall-clock instants; `relativeMs` is relative to per-event origins/clocks and would mis-compute across clocks. `relativeMs` stays the per-event *display/persistence* value the factory stamps.
2. **Output shape — typed `TurnMetrics` record vs `Dictionary<string, double?>`?** My default vote: **typed `TurnMetrics` record** (universal + cascade + realtime fields, all `double?`). Matches the project's record-heavy style, is discoverable, and maps cleanly to `ModeSummary` fields B.7 averages. (`double?` chosen over `long?` because session averages in `ModeSummary` are already `double?`.)
3. **`relativeMs` for an event before its origin — clamp to 0 or preserve negative?** My default vote: **clamp to 0 in the factory** (a single event's `relativeMs` is relative to a *same-clock* origin and should be monotonic; a negative display value is nonsense). Note: this clamp is the **factory's** single-event rule only — the **aggregator does NOT clamp** cross-clock metric pairs (Q1/test 5), where a small negative from skew is legitimate and disclosed.
4. **`speech_end_to_first_audio_ms` "first output-audio event" selection.** ARCH-013 defines it as "first output-audio event (or playback start) − `turn.recording.stopped`." A turn is either cascade or realtime, so the present event is `tts.first_audio` **or** `realtime.first_audio_delta`. My default vote: **prefer whichever output-audio event is present** (`tts.first_audio` for cascade, `realtime.first_audio_delta` for realtime); if neither, fall back to `playback.started`; if none, `null`.
5. **DI registration scope — register now or defer to B.4?** My default vote: **register `IClock`/factory/aggregator in `Program.cs` now.** The handoff scoped IClock's DI registration to B.3 (its first consumer is the factory); B.4 then resolves the trio for orchestration. Reachability is "available-in-DI, entry-point consumer = B.4" — name it, don't treat it as a silent gap.
