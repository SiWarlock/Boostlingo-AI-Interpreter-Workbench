# /tdd brief — error_sanitizer (B.8) — SAFETY

## Feature
`ErrorSanitizer` — the boundary that turns any internal/provider error (an arbitrary `Exception`, a `ProviderError`, or a `Result.Error` string) into a **safe, normalized `UiError`** that crosses the wire: **no stack traces, no secrets, no raw provider payloads** — only a fixed-safe message + a normalized `<stage>.<reason>` code, with `Retryable` (+ HTTP status) preserved. The **original is logged server-side only**. Introduces the backend `UiError` record (mirror of the ARCH-007 TS `UiError`). This slice **enforces safety invariant #4**.

> **SAFETY slice** — invariant #4 (root `CLAUDE.md` Key safety rules: "Provider errors are sanitized before reaching the UI … only a normalized `ProviderError`/`UiError` with a safe message + code (ARCH-018/ARCH-019). Enforced by `ErrorSanitizerTests`."). **Own commit, never bundled; mandatory `security-reviewer` pass at Step 8.**

## Acceptance criteria (what "done" means)
- [ ] Backend `UiError` record mirrors the ARCH-007 TS shape: `Code`, `SafeMessage`, `Stage?`, `Retryable`, `TurnId?` (serialized camelCase via `JsonDefaults`). HTTP status is preserved **server-side** for B.9's handler without polluting the wire body (→ 2.5 Q1).
- [ ] `ErrorSanitizer.Sanitize(Exception, …)` → `UiError` with a **fixed-safe message** + a generic code (`internal.error`), `Retryable=false`, status 500 (server-side) — the exception's `Message`/stack appear **nowhere** in the `UiError`.
- [ ] `ErrorSanitizer.ToUiError(ProviderError, turnId?)` → `UiError` carrying the (already-safe) `Code`/`SafeMessage`/`Stage`/`Retryable` (+ status server-side); the `Provider` field is **not** present on `UiError`.
- [ ] `ErrorSanitizer.SanitizeResult(error-code, …)` (or equivalent) maps a failed `Result`/`Result<T>` to a `UiError` **without echoing `Result.Error`** — a fixed-safe message per code; the raw error string never appears in the `UiError`.
- [ ] **The original (full) error is logged server-side** (injected `ILogger`) at the sanitize point — so diagnosis is possible while the client sees only the safe `UiError` (→ 2.5 Q2).
- [ ] **Safety pin:** an input whose text contains an API-key substring (`sk-…` / `ek_…`) and a stack trace produces a `UiError` whose `SafeMessage` (and every serialized field) contains **neither**.
- [ ] All unit tests in `AiInterpreter.Tests/ErrorSanitizerTests.cs` pass. `/preflight` clean. **Mandatory `security-reviewer` pass at Step 8.**

## RED test outline (Step 2) — `AiInterpreter.Tests/ErrorSanitizerTests.cs`
THIN volume, **strong safety pins** (tests 1 + 4 are the invariant centerpiece — write them first).

1. **`sanitize_exception_strips_secret_and_stack`** (SAFETY #4) — `new Exception("auth failed for sk-SENTINEL-KEY-123\n  at Foo.Bar()")` → `UiError.SafeMessage` contains **neither** `sk-SENTINEL-KEY-123` **nor** `at Foo.Bar`/stack text; assert across every serialized field (serialize the `UiError` and scan the JSON). `Code = internal.error`, `Retryable=false`.
2. **`sanitize_exception_generic_message`** — an arbitrary exception → a fixed safe message (e.g. "An unexpected error occurred.") + `internal.error` + status 500 (server-side). Not the exception text.
3. **`provider_error_projects_to_uierror`** — `ProviderError("deepgram","stt","stt.rate_limited","…retry shortly",true,429)` → `UiError(code="stt.rate_limited", safeMessage=…, stage="stt", retryable=true)` with status 429 preserved server-side; the serialized `UiError` JSON does **not** contain `"provider"` / `deepgram`.
4. **`result_error_string_not_echoed`** (SAFETY) — `SanitizeResult("persistence.failed", Result.Failure("persistence.failed: C:\\secret\\sk-abc\\session.json"))` → `UiError.SafeMessage` (and all fields) contain **neither** the path **nor** `sk-abc`; fixed-safe message; `code="persistence.failed"`.
5. **`retryable_and_status_preserved`** — a retryable provider error and a non-retryable one → `UiError.Retryable` matches each; the server-side HTTP status round-trips (429 vs 500).
6. **`original_logged_server_side`** — feed a secret-bearing exception with a captured/fake `ILogger` → the logger received the **original** (full) detail (server-side diagnosis intact) while the returned `UiError` is safe. Pins "log original server-side only."
7. **`turn_id_carried_when_supplied`** *(if cheap)* — `turnId` flows into `UiError.TurnId` when supplied; null/omitted otherwise.

## Things to flag at Step 2.5
1. **HTTP-status preservation vs the TS wire shape.** The TS `UiError` (ARCH-007) has **no `httpStatusCode` field** — the HTTP status is the response status line, set by B.9's handler. My default vote: the backend `UiError` mirrors the TS shape exactly for the body **and** carries a server-only `[JsonIgnore] int? HttpStatusCode` (null → 500 at the handler) so B.9 can set `response.StatusCode` without polluting the body. Alternative: `Sanitize` returns a `(UiError body, int status)` pair. Default: the `[JsonIgnore]` field (one type; body matches TS; same pattern as `Result.Error`).
2. **Injectable + logging.** My default vote: `ErrorSanitizer` is an **injectable singleton** taking `ILogger<ErrorSanitizer>`; it logs the original (full message + stack) at the sanitize point and returns the safe `UiError`. (Differs from the static `ProviderErrorMapper` because logging needs a logger.) Flag if you'd rather keep it static + log at the caller.
3. **Safe-by-construction, not scrub.** My default vote: the safe message is ALWAYS a **fixed string per code**; the sanitizer **never** interpolates `ex.Message`/`Result.Error`/stack into the output (lesson §5 principle). NO regex secret-scrubbing of the original (brittle). The tests prove safety by injecting a secret + asserting absence. Confirm.
4. **Division of labor with `ProviderErrorMapper`.** My default vote: the sanitizer **projects** an existing `ProviderError` → `UiError` and owns the **generic** `Exception` boundary + `Result.Error`; it does **not** re-implement the ARCH-012 HTTP-status table (the mapper owns that — provider-stream exceptions go through `ProviderErrorMapper` first, then project). Avoids two copies of the table.
5. **`UiError` placement.** My default vote: `Security/UiError.cs` (co-located with the sanitizer that produces it). Alternatives: `Common/` or next to `ProviderError` in `Providers/Abstractions/`. Flag your preference.
6. **DI registration.** My default vote: register `ErrorSanitizer` (singleton) now — consumers B.9/C.4. Consistent with prior seams. Flag if you'd defer to B.9.
