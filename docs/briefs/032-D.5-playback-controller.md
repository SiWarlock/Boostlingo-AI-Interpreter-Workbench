# /tdd brief — playback_controller

## Feature
The playback controller: streamed cascade TTS audio → progressive `MediaSource` playback (mp3), with a **fallback** to an assembled-blob `HTMLAudioElement` if MSE append fails; stamps `playback.started` on the audio element's `playing` event; **no overlapping playback**. Wires the D.4a/D.4b cascade client's `onAudio` callback (left a no-op in D.4b) to this controller — closing that foundation-ahead gap.

## Use case + traceability
- **Task ID:** D.5
- **Architecture sections it implements:** `ARCH-030` (§4 Audio playback — MSE progressive + blob fallback; stamp `playback.started` on `playing`; avoid overlap), `ARCH-013` (the `playback.started` latency marker — the only browser-side TTS-output timing, for cross-mode comparability).
- **Related context:** D.4a's `onAudio({contentType,seq,base64})` callback + the store's `appendLatencyEvent` (D.4a). D.4b left the client singleton's `onAudio` a no-op with a "D.5 wires playback" note (+ the settable-delegate suggestion). This slice builds the controller + wires it in.

## TDD scope (read first — mostly manual-smoke, like D.3)
**Test-first (deterministic, pure):**
- `decodeBase64Audio(base64) → Uint8Array` — decode a base64 `audio` chunk to bytes (for MSE append / blob assembly). Pure.
- The **no-overlap guard** — a small state fn/object: starting a new playback stops/resets the prior (at most one active). Testable as pure state if extracted from the element wiring.
- The `playback.started` **latency-event construction** — given the `playing`-event timestamp, build the `LatencyEvent` (`name:'playback.started'`, `stage:'playback'`) handed to `store.appendLatencyEvent`. Pure builder.

**Exempt (manual-smoke / ARCH-020):** `MediaSource`/`SourceBuffer` append + the `HTMLAudioElement` lifecycle + the actual audio output + the `playing`-event wiring. The controller is a browser-API shell over the pure helpers (the D.3 pattern).

## RED test outline (Step 2)
1. **`decode_base64_audio`** — Asserts: a known base64 string → the exact byte sequence (`Uint8Array`); empty → empty. Why: chunk decode correctness (MSE append / blob assembly).
2. **`no_overlap_guard_resets_prior`** — Asserts: starting playback while one is active stops/resets the prior (the guard reports a single active playback; a `reset()` clears it). Why: ARCH-030 no-overlap.
3. **`playback_started_built_once`** — Asserts: the `playback.started` `LatencyEvent` is built with `name:'playback.started'`/`stage:'playback'` and is stamped once (a second `playing` event for the same turn doesn't double-stamp). Why: ARCH-013 audible-start marker, once per turn.

> The `MediaSource`/`SourceBuffer`/`HTMLAudioElement` wiring + the MSE→blob fallback path are manual-smoke (browser internals). If a pure fallback-decision seam emerges (e.g. "after N append failures → blob"), flag at Step 2.5.

## Things to flag at Step 2.5
1. **Pure-helper extraction.** Default vote: extract `decodeBase64Audio` + the no-overlap guard + the `playback.started` builder (the testable seams); the MSE/element wiring stays manual-smoke.
2. **`onAudio` wiring — settable delegate vs reconstruct.** Default vote: **a settable delegate** on the client singleton (`setAudioSink(fn)`; the singleton's `onAudio` calls the current sink, default no-op) — D.5 calls `setAudioSink(playbackController.enqueue)`. Avoids reconstructing the singleton (per the D.4b note).
3. **MSE vs blob fallback trigger.** Default vote: **MSE first; on the first `appendBuffer` throw (or `MediaSource` unsupported), switch to blob assembly** → `HTMLAudioElement` (ARCH-030). The switch is manual-smoke; the decode/assembly bytes are pure.
4. **`playback.started` timing.** Default vote: stamp on the **`playing`** event (audible start, ARCH-030/013), once per turn — not on `enqueue`/first-chunk (which is arrival, not audible start).
5. **No-overlap mechanism.** Default vote: a new turn's `start()`/first `enqueue` calls `reset()` on the prior (stop the element / detach the MediaSource) before starting — single active playback.
