## RED test outline (Step 2)
**`DomainSerializationTests.cs`** (serialize/deserialize directly with the shared options — no host):
1. **`enums_serialize_as_camelCase_strings`** — `InterpretationMode.Realtime`→`"realtime"`, `LanguageCode.En`→`"en"`, `LatencyStage.Stt`→`"stt"`, `ClockSource.Server`→`"server"`. Why: ARCH-009 JSON shows lowercase string enums, not ints.
2. **`session_serializes_camelCase_matching_ARCH-009`** — a populated `InterpretationSession`/`SessionConfig`/`ProviderProfile` serializes with camelCase keys (`sessionId`, `startedAt`, `currentMode`, `providerProfile`, `realtimeModel`, …) matching the ARCH-009 `POST /api/sessions` example. Why: ARCH-009/016 contract.
3. **`latencyEvent_serializes_matching_ARCH-013_example`** — `{name, stage, timestamp, relativeMs, clockSource, metadata}` camelCase; `stage` as string. Why: ARCH-013 schema.
4. **`dateTimeOffset_is_iso8601`** — `StartedAt` serializes as ISO-8601 (`2026-05-28T15:30:00...Z`/offset). Why: ARCH-009 timestamp format.
5. **`round_trip_preserves_equality`** — serialize→deserialize a fully-populated `InterpretationSession` (with turns, transcripts, latency events, cost, errors) → equal to original. Why: persistence round-trip (B.7 depends on this).
6. **`nullable_fields_behave`** — `EndedAt=null`, `Summary=null`, `CostEstimate=null` serialize per the example convention (confirm omit-vs-null at Q1); empty lists serialize as `[]`. Why: ARCH-016 persisted shape.

**`ClockAndResultTests.cs`:**
7. **`system_clock_returns_utc_now`** + **`fake_clock_returns_fixed_time`** — `IClock` abstraction is injectable + deterministic.
8. **`result_success_and_failure`** — `Result` carries value on success, error on failure; no value-on-failure access.

## Things to flag at Step 2.5
1. **Nullable serialization — omit vs explicit `null`.** ARCH-016 persisted examples: do absent optionals omit the key or write `null`? My default vote: **`DefaultIgnoreCondition = WhenWritingNull`** (omit nulls) — matches typical compact JSON; but verify against the ARCH-016 example. If the examples show explicit `null`, drop the ignore condition. Confirm against the doc.
2. **Where the shared `JsonSerializerOptions` lives + its name.** My default vote: a static `Common/JsonDefaults.cs` exposing a configured `JsonSerializerOptions` (camelCase + `JsonStringEnumConverter(JsonNamingPolicy.CamelCase)`), reused by A.5 (`Program.cs` `AddJsonOptions`/`ConfigureHttpJsonOptions`) + B.7 persistence. Single source so API + persisted JSON can't diverge.
3. **File placement: `ProviderError` + enums.** ARCH-006 places `ProviderError` in `Providers/Abstractions/ProviderErrors.cs` and the domain model in `Sessions/SessionModels.cs`. My default vote: **follow ARCH-006** — `ProviderError` in `ProviderErrors.cs`; all other enums + records in `SessionModels.cs`. (B.1 later adds the provider *interfaces* + event types alongside `ProviderErrors.cs`.)
4. **Test file granularity.** One `DomainSerializationTests.cs` + one `ClockAndResultTests.cs`, or fold Clock/Result into the scaffold test? My default vote: **two focused files** as outlined.
5. **`Result` shape.** Generic `Result<T>` with `IsSuccess`/`Value`/`Error`, or a non-generic + generic pair? My default vote: a minimal `Result<T>` (+ maybe a non-generic `Result` for void ops) — keep it small; later seams (cascade, providers) will tell us what they need. Don't over-build.
