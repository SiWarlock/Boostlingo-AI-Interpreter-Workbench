## Things to flag at Step 2.5
1. **Flat-env→section bridge (load-bearing).** ARCH-028 env vars are flat (`DEEPGRAM_API_KEY`); A.2 Options bind from PascalCase sections (`Deepgram:ApiKey`). My default vote: **an explicit mapping layer in `Program.cs`** — `AddInMemoryCollection` (or a small custom source) mapping each ARCH-028 flat var → its `Section:Property`, *then* `Configure<T>(GetSection(SectionName))`. Keeps A.2's section-binding + tests intact. (Alt: the `Deepgram__ApiKey` double-underscore convention — but ARCH-028 documents flat names, so map them.)
2. **`/api/health` placement.** My default vote: **minimal-API `app.MapGet("/api/health", …)`** in `Program.cs` — simplest; no controller needed yet. The real `ConfigController` (`/api/config`) is B.9. (Or a thin `ConfigController` with only health now — your call; minimal-API is lighter.)
3. **Pricing loader DI shape.** My default vote: **load once at startup** (`PricingLoader.Load(PRICING_CONFIG_PATH)`) and register the `Result<PricingOptions>` (or a small `IPricingProvider` wrapping it) as a **singleton** — B.5's `CostEstimator` consumes it. Host starts regardless of load success.
4. **WAE scope (resolves the Carry-forward).** My default vote: **keep Api-only** (production-code strictness) and **relax on the Tests project** — test SDKs/xUnit analyzers generate warnings that under WAE are pure friction; this is the conventional posture and matches what's shipped in A.1–A.4. So: **no `Directory.Build.props`; close the carry-forward as "decided: Api-only WAE."** (If you feel strongly about solution-wide, flag it — but I lean Api-only.)
5. **Template reconciliation (A.1 leftover).** Set port **5179** + fix `launchSettings` (drop the 5243/7074 template ports). Swashbuckle/OpenAPI + the `swagger` `launchUrl`: my default vote: **keep OpenAPI/Swagger** (low cost, genuinely useful for dev/demo API exploration) and point the dev `launchUrl` at `/swagger` or `/api/health`. (Remove only if you want a leaner host — flag.)
6. **Exercising the JSON-contract test (#2) pre-B.9.** No real domain endpoint exists until B.9. My default vote: assert the JSON contract via the health route plus a **test-only minimal endpoint** registered in the test host that returns a domain enum/record, OR assert `JsonDefaults`-configured options are the ones the pipeline uses. Pick the lightest that genuinely proves `Apply` is wired — flag your approach.
