# /tdd brief — recording_controls_and_cascade_wiring

## Feature
`RecordingControls` (Start/Stop + capture timer + status) and the **recording orchestration** that wires the whole cascade streaming turn together: Start → `createTurn` → `beginTurn` → open the cascade WS → stream the capture controller's PCM frames; Stop → stop capture + send `stop`. This is **D.4b** — it makes the cascade recording path reachable end-to-end (closing the D.1 `createTurn`, D.3 capture, and D.4a client foundation-ahead-of-callers gaps).

## Use case + traceability
- **Task ID:** D.4 (slice b)
- **Architecture sections it implements:** `ARCH-007` (RecordingControls + the enforced recording-transition table), `ARCH-011` (the streaming cascade turn — capture → WS → live transcripts), `ARCH-009` (`POST …/turns` create + the WS start).
- **Related context:** Wires three landed pieces — D.1 `sessionsApi.createTurn` + the store, D.3 `audioCaptureController` (`startStreaming({onFrame,onError}) → {sampleRate, encoding:'linear16', stop()}` + `recordBlob()`), D.4a `cascadeStreamClient` (`start/sendFrame/stop`, `dispatchCascadeMessage`, the 6 streaming store actions). The realtime path (E) and playback (D.5) are out of scope; `onAudio` is stubbed here (D.5 wires playback).

## Acceptance criteria
- [ ] Pure transition helpers (added to `web/src/state/selectors.ts`): `canStartRecording(state)` (true when `sessionStatus` is `active`/`readyForTurn` AND `turnStatus` is `ready`/`completed`/`failed`), `canStopRecording(state)` (true when `turnStatus === 'recording'`) — the ARCH-007 transition table.
- [ ] Recording orchestration (DI'd, testable — see Step-2.5 Q2): `startRecording(deps)` → `sessionsApi.createTurn(sessionId)` → `store.beginTurn({turnId, mode, direction})` → open the WS (`cascadeStreamClient.start({sessionId, turnId, direction, encoding:'linear16', sampleRate, translationModel, ttsVoice})`) → start capture and pipe `onFrame → client.sendFrame`, `onError → store.failTurn`. `stopRecording(deps)` → `capture.stop()` + `client.stop()` → `store.setTurnStatus('processing')`. On a `createTurn` `ApiError` → `store.addError` + abort (no orphaned capture/WS).
- [ ] **Frame sequencing handled** (Step-2.5 Q1): frames captured before the WS `open` are NOT dropped or sent on a CONNECTING socket — resolved per the chosen approach (default: a small pre-open send-queue in the client, flushed on open).
- [ ] `RecordingControls` (`web/src/components/RecordingControls.tsx`) renders Start (disabled unless `canStartRecording`) / Stop (disabled unless `canStopRecording`), a capture timer, and the current `turnStatus`. Calls `startRecording`/`stopRecording`. Renders only from the store + selectors — no transport/capture internals leak into the component (clean separation; the orchestration owns the wiring).
- [ ] `App.tsx` mounts `RecordingControls`. After D.4b, a cascade turn runs capture → WS → live store updates (transcripts/latency/cost) end-to-end (playback + panels are D.5/D.6).
- [ ] All unit tests in `selectors.test.ts` (transition helpers) + the orchestration tests pass; `/preflight` clean.

### recording orchestration (`recordingActions.test.ts` — mock deps)
3. **`start_recording_happy_path`** — `createTurn` → `beginTurn(turnId,…)` → `client.start(startParams with sampleRate+turnId)` → capture started + `onFrame` piped to `sendFrame`. 4. **`start_recording_createTurn_error_aborts`** — `createTurn` rejects `ApiError` → `store.addError`, NO `beginTurn`/`client.start`/capture started (no orphaned resources). Why: failure path / no leak.
5. **`frames_before_open_are_not_lost`** — a frame emitted before WS `open` is queued + flushed on open (or per the chosen approach) — never `send` on a CONNECTING socket. Why: Step-2.5 Q1 (the real integration edge).
6. **`stop_recording_stops_capture_and_socket`** — `stopRecording` → `capture.stop()` + `client.stop()` + `store.setTurnStatus('processing')`. Why: stop flow; the `done` message later drives `completeTurn` (D.4a).

## Things to flag at Step 2.5
1. **Frame sequencing before WS open (the real integration edge).** Capture's `sampleRate` is known synchronously (AudioContext), but frames begin flowing immediately while `cascadeStreamClient.start` opens the socket async (start frame sent on `onopen`). Options: **(A)** a small **pre-open send-queue in the client** — `sendFrame` queues if `readyState!==OPEN`, flushes on open (a ~5-line touch-up to the committed D.4a client; the client owning its readyState is the right home); **(B)** the orchestration buffers frames until an `onOpen` signal then flushes; **(C)** sequence capture-after-open (needs the rate before start — awkward). Default vote: **(A)** — cleanest home, smallest surface, and the client is the natural owner of socket-readiness. Touching the committed D.4a client in the same phase is fine.
2. **Orchestration location.** `web/src/state/recordingActions.ts` (new, parallel to D.2's `sessionActions.ts`) vs extending `sessionActions.ts`. Default vote: **a new `recordingActions.ts`** — keeps the turn-recording orchestration separate from session start/end; both are DI'd + tested against the real store + mocked deps.
3. **`onAudio` in D.4b.** Playback is D.5. Default vote: the orchestration accepts an `onAudio` dep **defaulting to a no-op**; D.5 wires the real playback controller. (Don't drop the audio frames silently in a way that hides a D.5 gap — a no-op with a "// D.5 wires playback" comment.)
4. **`stopRecording` target status.** Default vote: `turnStatus → 'processing'` on stop (waiting for the trailing finals + the `done` message, which `completeTurn` turns into `'completed'`). The ARCH-007 table forbids a new recording / mode-switch during `processing`.
5. **`createTurn` before `start`.** Default vote: **yes** — the WS `start` frame needs the backend-minted `turnId`, so `createTurn` (HTTP) precedes the WS open; a `createTurn` failure aborts before any capture/WS resource is created.
