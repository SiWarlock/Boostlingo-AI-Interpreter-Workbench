# /tdd brief — realtime_mode_switch_teardown

## Feature
Flow-G frontend mode-switch: when the user switches mode **away from realtime** (realtime → cascade), tear down the realtime connection (release the pc, mic track, data channel, `<audio>`) so it doesn't linger — preventing a double-mic / hot-realtime-connection while cascade runs. Wired to the mode toggle.

## Use case + traceability
- **Task ID:** E.5b (the mode-switch half of E.5; E.5a shipped the lifecycle `a09c15e`). **Scoped TIGHT** — see "Deferred / re-sequenced" below.
- **Architecture sections it implements:** `ARCH-010` (§7 — "torn down on End Session or **mode-switch-away**"), `ARCH-017` (Flow G — mode switch between turns), `ARCH-007`.
- **Related context:**
  - E.5a shipped (`a09c15e`): `realtimeConnectionManager` with `ensureConnected()` + **`teardown()`** (close DC/pc, stop tracks, release stream, detach `<audio>`, reset latches). E.5b **reuses `teardown()`** on mode-switch-away (it's already wired into SessionSetup End; E.5b adds the mode-switch trigger).
  - D.2 built the mode selector + `currentMode` in the store; E.4b's RecordingControls already dispatches turns by `currentMode`. So **mode-switch already routes turns correctly** — the gap E.5b closes is the realtime **connection teardown** on switch-away (else the realtime pc + mic stay live while cascade captures its own mic → two live mic streams; ARCH-010 teardown discipline).
  - The mode toggle component (confirm at Step 1 — `ModeToggle` and/or the `SessionSetup` mode selector, D.2).

## Acceptance criteria (what "done" means)
- [ ] On `currentMode` change **realtime → cascade** (a mode-switch-away), `realtimeConnectionManager.teardown()` is invoked (release pc/mic/DC/`<audio>`, reset latches) — so no realtime mic/connection lingers during cascade.
- [ ] On `currentMode` change **cascade → realtime**, NO premature teardown — realtime (re)connects lazily on the next `startTurn` (E.5a `ensureConnected`).
- [ ] A mode change that is NOT a switch away from realtime (e.g. realtime→realtime no-op, or direction-only change) does NOT tear down.
- [ ] The teardown-on-switch is invoked from the **real mode-toggle path** (the component the user clicks), not test-only (Step 7.5).
- [ ] All unit tests pass; `/preflight` clean.
