## Feature
`GET /api/sessions/{id}` falls back to a by-id DISK read when the session isn't in the in-memory store: add
`SessionPersistenceReader.ReadById(id)` (enumerate `SESSION_DATA_DIR`, return the persisted `InterpretationSession`
whose `SessionId` matches, degrade-safe) and have `SessionService.Get` try in-memory FIRST, then disk — so a past
session (after a server restart or eviction) returns `200` + its full detail instead of a `404`.

## Design (transparent in-memory→disk fallback; the by-id reader degrades like the list reader)
- **`SessionPersistenceReader.ReadById(string id)`** → `Result<InterpretationSession?>` (or `InterpretationSession?`
  — Q1 picks how the service consumes it):
  - **Validate the id FIRST** (`^[A-Za-z0-9_-]+$`, reuse §11) — an invalid id returns "not found" (null) **without
    touching the FS** (Q3 — defense-in-depth + load-bearing if a future optimization ever constructs a path from
    the id; a `../` / non-allowlist id never reaches enumeration).
  - Enumerate `SESSION_DATA_DIR` for `session_*.json` (`TopDirectoryOnly`, no recursion — §11 read-side) and
    return the FIRST file whose **deserialized `SessionId == id`** (reuse `TryReadFile` — same single-open
    size-guard + filtered catch). Degrade PER-FILE: a corrupt/oversize file is skipped (§3/§35) — if the ONLY
    matching-named candidate is corrupt → treat as not-found (null), never a throw/500 (Q5).
  - Missing dir → not-found (null). A `dataDir`-where-file misconfig → the §35 wholesale-failure semantics (let
    the service decide null vs surfaced error — Q1).
- **`SessionService.Get`** — try **in-memory FIRST** (`SessionStore.Get` — the freshest copy of a LIVE session),
  then **disk** (`reader.ReadById`) when in-memory misses; return null only when neither has it. **In-memory wins
  when both exist** (Q4 — a live session's in-memory state is fresher than its last-persisted snapshot). The
  controller's null→404 mapping is unchanged.
- **Invariant #3 (no secret/audio) holds structurally** — `ReadById` deserializes the same `InterpretationSession`
  the writer wrote (065's read is already sentinel-covered). **Keep the `SessionPersistenceTests` sentinel current**
  + assert the by-id disk-read response stays clean (no `sk-`/`ek_`/audio) — same meta-rule as §29/§35.

## Acceptance (RED-first; deterministic — temp-dir fixtures + `WebApplicationFactory`, never a live FS dependency)
- [ ] **`ReadById` returns the matching persisted session** from a temp dir of 2–3 fixture `session_*.json` (match
      on `SessionId`, not filename). An id with NO matching file → not-found (null). A missing dir → not-found.
- [ ] **Degrade:** a corrupt/oversize file in the dir is SKIPPED (reuse `TryReadFile`); if the would-be match is
      corrupt → not-found (null), never a throw. Pin it.
- [ ] **⭐ Path/id safety:** an invalid id (`../…`, a non-allowlist char) → not-found **without enumerating the FS**
      (assert via a spy / a sentinel file outside the dir staying unread, or a no-enumerate guard). The reader never
      reads outside `SESSION_DATA_DIR`. (Safety rule #5.)
- [ ] **`SessionService.Get` fallback arms:** in-memory present → returns the in-memory session (in-memory WINS even
      when a disk copy exists — pin with both populated); in-memory absent + on-disk → returns the disk session;
      neither → null. Pin all three.
- [ ] **`GET /api/sessions/{id}` wire** (`WebApplicationFactory`, seed a temp `SESSION_DATA_DIR` + an EMPTY in-memory
      store): a persisted-but-evicted id → `200` + the full `InterpretationSession`; an unknown id → sanitized
      `404 session.not_found` (unchanged); no path/exception/secret leak on either arm.
- [ ] **Sentinel maintained:** the disk-read `GET /{id}` response carries no `sk-`/`ek_`/audio (round-trip clean);
      `BuildFullSession` stays current.
- [ ] `/preflight` (backend) clean; full `dotnet test` green; build 0W/0E.

## Things to flag at Step 2.5
1. **Fallback location — transparent in `SessionService.Get` (my vote) vs a separate `GetPersisted`/separate route.**
   My vote: **transparent in `Get`** (in-memory first, disk second) → the controller + the frontend stay
   byte-identical; `GET /{id}` simply stops 404ing for evicted sessions. A separate route adds a contract surface
   the FE drill-in doesn't need. Confirm.
2. **`ReadById` strategy — enumerate + match on deserialized `SessionId` (my vote) vs construct the filename from
   the id.** My vote: **enumerate + match** — the filename embeds `StartedAt` (not the bare id) so it can't be
   constructed; enumeration reuses `TryReadFile` + degrades per-file. (Sessions are few; O(n) deserialize is fine
   for the single-user threat model.) Confirm.
3. **⭐ id validation pre-FS — gate `^[A-Za-z0-9_-]+$` before any enumeration (my vote) vs rely on dir-scoped
   enumeration alone.** My vote: **gate first** (reuse §11 `IsValidSessionId`) — defense-in-depth (safety rule #5)
   and load-bearing if a future optimization narrows by filename fragment. An invalid id → not-found, no FS touch.
   Confirm.
4. **Precedence when a session is BOTH in-memory and on disk — in-memory wins (my vote).** My vote: **in-memory
   first** — a live session's in-memory state is fresher than its last-persisted snapshot (persist happens at
   `/complete`/`/end`). Confirm.
5. **Corrupt-only match — degrade to not-found/404 (my vote) vs surface a 500.** My vote: **not-found** — a corrupt
   file is effectively unreadable; a drill-in on one bad file shouldn't 500. Skip-and-log (§3/§35). Confirm.
