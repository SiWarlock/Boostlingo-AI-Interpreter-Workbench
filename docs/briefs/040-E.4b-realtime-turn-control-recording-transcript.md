# /tdd brief — realtime_turn_control_and_wiring

## Feature
The realtime turn controller (manual VAD-off Start/Stop) + its wiring into RecordingControls (so realtime turns are **reachable from the real UI**) + the TranscriptPanel "source unavailable" note. This is the E.4 wiring slice — it makes the realtime path work end-to-end in the browser (modulo connection-lifecycle robustness, which is E.5).

## Use case + traceability
- **Task ID:** E.4b (the turn-control + UI half of E.4; E.4a shipped the data path `7b63fc4`).
- **Architecture sections it implements:** `ARCH-010` (§7 — manual turn control: `turn_detection=null`, `input_audio_buffer.clear`/`commit`, `response.create`; the metrics split), `ARCH-013` (latency), `ARCH-007` (clean separation: components dispatch intents; the controller + client own the transport).
- **Related context:**
  - E.4a shipped (`7b63fc4`): `createRealtimeEventSink({store, clock})` + `sessionsApi.appendTurnEvents`. E.4b's controller creates a sink per turn, feeds it the client's server-event stream, and calls `appendTurnEvents` at turn-end.
  - E.3 shipped (`87d77d0`): `createRealtimeWebRtcClient` (the handshake shell) + `parseRealtimeEvent`/`normalizeRealtimeEvent`. E.4b extends the client with the DC send/receive surface (below) + wires parse→normalize→sink.
  - **The cascade precedent — web §11 (recording orchestration):** "Wire a recording turn via a DI'd controller (createTurn→store→capture→transport, tested vs the real store + mocked deps); guard concurrent starts with an `inFlight` flag." E.4b's realtime controller is the direct analogue (the cascade `recordingController` / D.4b is the structural model).
  - D.4b wired RecordingControls to dispatch the cascade turn; E.4b adds the **realtime branch** (dispatch by `UiSessionState.currentMode`).
  - ARCH-010 manual turn control: with `turn_detection=null`, the mic track streams continuously (E.3 `addTrack`); the turn is **delimited** by `input_audio_buffer.clear` (Start) and `input_audio_buffer.commit` + `response.create` (Stop) — there is NO per-turn mic start/stop.

## Acceptance criteria (what "done" means)
- [ ] `createRealtimeTurnController({ store, client, api, clock })` (or equivalent DI'd shape):
  - **`startTurn()`** → `api.createTurn(sessionId)` → `store.beginTurn({ turnId, mode:'realtime', direction })` → ensure the client is connected (lazy `connect(...)` if not — E.5 hoists this to session-start) → create a per-turn sink (E.4a) → wire `client.onServerEvent = raw => { const e = normalizeRealtimeEvent(parseRealtimeEvent(raw)); if (e) sink.handle(e); … }` → `client.sendClientEvent({ type:'session.update', … turn_detection:null })` → `client.sendClientEvent({ type:'input_audio_buffer.clear' })` → stamp `turn.recording.started` (browser clock).
  - **`stopTurn()`** → `client.sendClientEvent({ type:'input_audio_buffer.commit' })` → `client.sendClientEvent({ type:'response.create' })` → stamp `turn.recording.stopped` (browser clock).
  - On the sink's `responseDone` (turn finalized) → `api.appendTurnEvents(sessionId, turnId, <the turn's accumulated latencyEvents>)` (the E.4a reporting trigger).
  - Concurrent-start guard (`inFlight`) — a 2nd `startTurn` while one is in flight is a no-op (web §11 concurrency edge).
- [ ] `client.sendClientEvent(obj)` (DC `send(JSON.stringify(obj))`) + a settable `onServerEvent` delegate (DC `onmessage` → delegate) added to `realtimeWebRtcClient` as **manual-smoke shell** surface (the DC send/receive is browser-internal; documented exempt). The controller's *use* of them is what's tested (against a mocked client).
- [ ] **RecordingControls** dispatches Start/Stop to the **realtime** controller when `currentMode==='realtime'` (and the cascade controller when `'cascade'` — existing). Realtime turns are now **reachable from the real UI** (Step 7.5 satisfied — the first real realtime entry point).
- [ ] **TranscriptPanel** renders source + target; when realtime input-transcription yields no source (PRD must-have 6 / ARCH-010), show an explicit **"source unavailable"** note (never silently hide the source row).
- [ ] All unit tests pass (`realtimeTurnController.test.ts` + the TranscriptPanel/RecordingControls tests). `/preflight` clean.

## Things to flag at Step 2.5
1. **Confirm the E.3 client's actual surface.** E.4b assumes a `connect(...)` + adds `sendClientEvent`/`onServerEvent`. Confirm what `createRealtimeWebRtcClient` already exposes (connect/ontrack) vs what E.4b adds — adjust the file plan if E.3 already stubbed a send/receive seam. My default: add `sendClientEvent` + `onServerEvent` here as shell surface.
2. **Lazy connect (E.4b) vs connect-at-session-start (E.5).** My default vote: **lazy `connect()` on first `startTurn`** (interim — gets the realtime turn working end-to-end now); E.5 hoists to session-start + persistent + teardown. Flag if you'd rather stub a connection seam E.5 fills.
3. **`appendTurnEvents` trigger point — on `responseDone` (sink) vs in `stopTurn`?** My default vote: **on `responseDone`** (the turn is finalized then — the events are complete; `stopTurn` only commits + asks for the response, which is still streaming). The controller observes `responseDone` via the sink/event stream.
4. **Slice size — keep TranscriptPanel here or split?** My default vote: **keep it** (it's small + PRD must-have 6 + the realtime turn isn't demoable without the source/target render). If at GREEN the slice exceeds ~160 lines, flag and we split TranscriptPanel into E.4c.
5. **"source unavailable" gating signal.** How does the panel know input-transcription is off? My default vote: **derive from state** — realtime turn + a completed/active turn with zero source segments → "source unavailable" (don't add a new flag if the absence is observable). Flag if a config signal (the mint's transcription-enabled) is cleaner.
