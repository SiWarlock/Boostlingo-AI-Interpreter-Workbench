# /tdd brief — realtime_event_sink_and_reporting

## Feature
The Realtime **data path**: a per-turn stateful event sink that maps E.3's `NormalizedRealtimeEvent`s into the store (transcripts by role + first-of-type latency stamps, browser-clock) and reports the turn's accumulated client events to the backend (`POST …/turns/{turnId}/events`). E.4b then wires turn control + the UI on top.

## Use case + traceability
- **Task ID:** E.4a (split from E.4 — "manual turn control + event mapping + metrics"; this slice is the data path, E.4b is turn control + RecordingControls + TranscriptPanel). **D.4a→D.4b precedent** (cascade split the streaming data path from the recording-controls wiring).
- **Architecture sections it implements:** `ARCH-010` (§7 — GA event mapping + the realtime metrics), `ARCH-013` (metrics/latency model), `ARCH-009` (§6 — `POST /api/sessions/{id}/turns/{turnId}/events`).
- **Related context:**
  - E.3 shipped (`87d77d0`): the pure `normalizeRealtimeEvent` → `NormalizedRealtimeEvent` union (`audioDelta` / `targetTranscriptDelta` / `sourceTranscriptDelta` / `sourceTranscriptCompleted` / `responseCreated` / `responseDone` / `error`). E.4a consumes that union — it does NOT re-parse GA events.
  - The store's streaming actions (D.4a, mode-agnostic): `appendTranscriptSegment(segment)`, `appendLatencyEvent(event)`, `failTurn(uiError)`, `completeTurn(turnId, status)`, `beginTurn({turnId, mode, direction})`. Realtime reuses these verbatim (the cascade WS drives them today).
  - The cascade precedent: the pure `dispatchCascadeMessage` router (web §9) + the store-normalizing transcript action (web §10). E.4a's sink is the realtime analogue — but **stateful** (per-turn first-of-type tracking), so it's a closure/factory, not a pure fn.
  - Metrics three-source model (web §13): realtime differs from cascade — **realtime reports its client events to the backend via `POST …/events`, and the backend computes the canonical top-level metrics** (cascade structurally can't, so the frontend computes those; realtime can, so it reports + reads `GET /summary`). E.4a is the "report" half.
  - Backend endpoint (B.9c-ii, shipped): `POST /api/sessions/{id}/turns/{turnId}/events`, body `AppendEventsRequest { Events: List<LatencyEvent> }` → wire `{ "events": [ …LatencyEvent ] }` (`[Required, MaxLength(500)]`); persists the events verbatim (with `clockSource`).

## Acceptance criteria (what "done" means)
- [ ] `createRealtimeEventSink({ store, clock })` returns a sink with `handle(event: NormalizedRealtimeEvent): void`, holding **per-turn** first-of-type state (a fresh sink per realtime turn). It maps:
  - `targetTranscriptDelta` → `appendTranscriptSegment({ role:'target', text, isFinal:false, … })`; **first** one → stamp `realtime.first_transcript_delta` (browser clock).
  - `sourceTranscriptDelta` → `appendTranscriptSegment({ role:'source', isFinal:false, … })`; `sourceTranscriptCompleted` → `{ role:'source', isFinal:true }`.
  - `audioDelta` → **stamp `realtime.first_audio_delta` (first only), browser clock — TIMING ONLY.** It writes **no** transcript and **no** audio payload to the store (invariant #3 — realtime audio plays via the WebRTC media track / E.3 `ontrack`, never through here; the delta's base64, if present, is discarded).
  - `responseDone` → `completeTurn(turnId, 'completed')` + stamp `turn.completed` (browser clock).
  - `responseCreated` → no store action (lifecycle marker; E.4b/E.5 may consume it).
  - `error` → `failTurn(uiError)` where `uiError = { code, safeMessage:<generic>, retryable, stage:'realtime' }` — the GA `error.code` is the code, the `safeMessage` is a fixed generic (never the raw GA message — E.3 already dropped it; E.5 builds the fuller `ProviderError` map).
- [ ] First-of-type stamps fire **exactly once per turn** (a 2nd `audioDelta`/`targetTranscriptDelta` does NOT re-stamp).
- [ ] `appendTurnEvents(sessionId, turnId, events: LatencyEvent[])` → `POST /api/sessions/{id}/turns/{turnId}/events` with `Content-Type: application/json`, body `{ events }` (matches the backend `AppendEventsRequest` wire shape); a failure surfaces as `ApiError(UiError)` via the shared `http` boundary (no raw-body leak).
- [ ] All unit tests in `web/src/realtime/realtimeEventSink.test.ts` + the extended `web/src/api/sessionsApi.test.ts` pass.
- [ ] `/preflight` clean.

## Things to flag at Step 2.5
1. **`audioDelta` is TIMING-ONLY — confirm + the WebRTC-emits-`output_audio.delta` uncertainty.** In **WebRTC** mode the model's audio arrives as a **media track** (E.3 `pc.ontrack` → `<audio>`), NOT as base64 over the data channel. ARCH-010's mapping assumes `response.output_audio.delta` events still arrive on the DC (as the first-audio TIMING marker) — but it's **smoke-pending** whether WebRTC emits them at all (they may be WebSocket-transport-only). My default: the sink stamps `realtime.first_audio_delta` **if** an `audioDelta` arrives (defensive); if WebRTC emits none, that stamp stays honest `n/a` and **`playback.started`** (the `<audio>` playing event, E.4b/E.5) is the reliable browser-side first-audio timing. Either way `audioDelta` writes no audio to the store (invariant #3). Flag this as a smoke-confirm.
2. **`appendTurnEvents` placement — `sessionsApi` vs `realtimeApi`?** The endpoint is session/turn-namespaced (`/api/sessions/…/events`), not realtime-namespaced. My default vote: **`sessionsApi`** — matches the `createTurn`/`endSession` namespace + keeps `realtimeApi` to the OpenAI-facing mint. (Only realtime *uses* it today, but the endpoint is generic.)
3. **Sink statefulness — fresh sink per turn vs a `reset()`?** My default vote: **a fresh `createRealtimeEventSink(...)` per realtime turn** (E.4b creates one when the turn begins) holding `firstAudio`/`firstTranscript` booleans — no `reset()` foot-gun, no cross-turn bleed. (Mirrors the per-turn nature of the cascade recording controller, web §11.)
4. **`error` → `UiError` mapping depth.** My default vote: **minimal** in E.4a — `{ code: <GA error.code>, safeMessage: <fixed generic>, retryable:false, stage:'realtime' }`; the fuller `realtime.session.disconnected`/`response.error` → `ProviderError` map is **E.5**. Just enough to `failTurn` honestly + no-leak.
5. **`segmentId` / `timestamp` provenance for realtime segments.** The store's `appendTranscriptSegment` expects a full `TranscriptSegment` (`segmentId`, `provider`, `timestamp`, `clockSource`). My default vote: the sink synthesizes `clockSource:'browser'`, `timestamp` from the injected `clock`, `provider:'openai-realtime'`, and a stable `segmentId` per role-stream (so deltas of the same role replace/extend per the store's §10 normalization). Flag if you'd source any of these differently.
