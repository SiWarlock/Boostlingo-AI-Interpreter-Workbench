# Styling brief — ui_baseline (H.1)

> **This is a STYLING brief, NOT a `/tdd` brief.** Visual styling is manual-smoke-exempt (root `CLAUDE.md` TDD posture — like mic/WebRTC). H.1 is verified by a **visual checklist**, not RED tests. The discipline that DOES apply: **clean-separation (ARCH-007)** — components still render only from the store; H.1 is **CSS + layout + minimal markup-for-styling only, NO logic changes** — and the existing unit/component suites must stay **green** (no contract/behavior regression).

## Feature
A first usability/visual pass: a global design-system/CSS baseline (tokens + app-shell layout + visual hierarchy) that makes the workbench **presentable and gives clear UX feedback** — most importantly **visible active-mode selection** (today the unstyled `ModeToggle` reads as "broken") + **session/turn status cues** + **legible panels** (Transcript / Metrics / Cost / Evaluation / Comparison). Enough to drive the demo confidently. Deeper polish (responsive, full a11y, motion) is H.2+ (later).

## Acceptance criteria (visual/UX pins — verified by manual smoke)
- [ ] **Design-system baseline:** CSS custom properties for a color palette, a spacing scale, and typography; a CSS reset/normalize; a coherent visual language applied app-wide (not per-component ad-hoc).
- [ ] **App-shell layout:** a clear structure — a header/title, the mode toggle + session controls, and the main content (the live panels grouped, the Evaluation + Comparison panels placed) — laid out (grid/flex), not a raw vertical stack of unstyled elements.
- [ ] **⭐ Visible active-mode selection (the headline gap):** the selected mode in `ModeToggle` is **unmistakably** visually distinct (style the `aria-pressed="true"` state — filled/highlighted/underlined). A user must instantly see which mode is active. _(This is the bug that made the toggle look broken.)_
- [ ] **Session + turn status cues:** `sessionStatus` (configured / active / ending / ended) and `turnStatus` (idle / recording / processing) render as **clear visual indicators** (a status pill/badge/dot) — the user can see the app's state at a glance. A recording turn looks active (e.g. a recording indicator).
- [ ] **Styled controls:** Start / Stop / End / Record buttons + the model/direction `<select>`s are styled, grouped, and have visible disabled/active states (the existing gating already disables them — make "disabled" look disabled).
- [ ] **`ErrorBanner` styled as an error** (the existing sanitized `errorCopy` is the content; H.1 makes it look like an error surface — color/icon/placement).
- [ ] **Legible panels:** Transcript (source/target, partial vs final distinguishable), Metrics (the latency numbers + `n/a`), Cost (the $/min), **EvaluationPanel** (the WER % / S·I·D·N + the "WER is STT-only" note as readable structure), **ComparisonSummary** (the by-mode + by-variant grid as a real table/cards, the `n/a`/"unavailable" states clear).
- [ ] **No logic/contract change:** components gain `className`s + minimal semantic wrappers only; the store, the API clients, the transports, and component *behavior* are untouched. The clean-separation invariant holds.
- [ ] **Suites stay green:** `npm run format:check && npm run lint && npm run typecheck && npm run test` clean — the markup-for-styling additions don't break the jsdom/Testing-Library component tests (they query by role/text; confirm). If a test queries structure that a wrapper changes, adjust the test minimally (a query, not the assertion) + note it.

## Design questions to settle (the Step-2.5-equivalent — answer before styling)
1. **(LOAD-BEARING) CSS approach.** Options: (a) **plain CSS** — global tokens/reset + co-located component `.css` imports; (b) **CSS Modules** (`*.module.css`, scoped class names, Vite-native); (c) a utility framework (Tailwind — adds a dep + config). My default vote: **(a)/(b) dependency-free** — a global tokens+reset+shell stylesheet **plus** CSS Modules for component-scoped styles (Vite supports both with zero deps; scoped + easy to expand in H.2). **Avoid Tailwind for H.1** (a new build dep + a different idiom — heavier than a demo baseline needs; can revisit in H.2 if wanted). Confirm.
2. **Status cues — style existing rendered state, or add a small status indicator?** If `sessionStatus`/`turnStatus` aren't currently rendered visibly anywhere, adding a small status badge is a **markup addition that reads store state the component already has** (no new logic). Default: **add a minimal status pill** (reads `useSessionState()` — the component already subscribes). Confirm this stays within "no logic change" (it's display-only).
3. **Scope discipline — how much markup restructuring?** Default: the **minimum** wrappers needed for layout/legibility (a panel container, a controls row, a status pill) — not a component rewrite. Keep diffs reviewable + the tests green. Confirm the bar: "CSS + the smallest markup that makes it legible," not "redesign the component tree."
4. **Visual direction.** Default: a clean, professional, **information-dense** workbench aesthetic (this is an instrumented evaluation tool, not a consumer app) — legible typography, clear data tables, restrained color used for *state* (active mode, recording, error). Confirm the vibe (the user said "look good with good UX" — lean utilitarian-but-polished, not flashy).
