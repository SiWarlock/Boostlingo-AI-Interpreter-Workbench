# /tdd brief — turn_lifecycle_endpoints (B.9c-ii)

## Feature
`SessionsController` **turn-lifecycle** routes (backend owns `turnId`): `POST /api/sessions/{id}/turns` (create empty turn), `POST …/turns/{turnId}/events` (append normalized `LatencyEvent`s), `POST …/turns/{turnId}/complete` (finalize the turn + best-effort persist). Defines the **`CompleteTurn` request DTO** (ARCH-009 underspecifies it — this slice realizes it; orchestrator documents it back). Completes the B.9 split (a/b/c-i/c-ii).

## Things to flag at Step 2.5
1. **`turnId` generation/ownership.** My default vote: `SessionStore.CreateTurn(sessionId, mode, direction)` owns id generation (consistent with `sessionId`), thread-safe under the gate; format `turn_<short-id>` (lowercase GUID segment, matches `^[A-Za-z0-9_-]+$`). The ARCH-009 `turn_001` is **illustrative** — a GUID segment avoids a per-session counter. Flag if you want sequential `turn_NNN`.
2. **`POST …/turns` body.** My default vote: **no request body** — the empty turn inherits `Mode` + `Direction` from `session.Config` (current mode at creation; Flow G mode-switch is recorded separately). Flag if you'd rather the request override mode/direction.
3. **`CompleteTurnRequest` shape (the spec-gap — needs your confirm).** My proposed shape (all optional, merged into the turn that already holds `/events` latency): `{ audioDurationMs?, transcripts?[], costEstimate?, werResult?, translationModelUsed?, ttsVoiceUsed?, status? }`. Rationale: the Realtime client (E.4) computes transcripts + metadata client-side and reports the finalization payload here; `/events` carries latency separately; cascade doesn't use this route. **Confirm the field set before GREEN** — I document it into ARCH-009. (Status defaults to `Completed`; a failed turn can pass `Failed` + an `Errors` entry per invariant #2.)
3b. **Transcripts via `/complete` vs `/events`.** My default vote: **`/complete` carries transcripts** (the client has them at turn end); `/events` stays latency-only (ARCH-009's events example is `LatencyEvent`-shaped). Flag if you read ARCH-009 as transcripts-via-events.
4. **`/complete` persist semantics.** My default vote: **best-effort** (ARCH-016 per-turn) — always mark Completed in memory; a write failure → 200 + `persistence.failed` `persistenceWarning` (reuse the B.9c-i `EndSessionResponse` pattern or a `CompleteTurnResponse{turn, persistenceWarning?}`), never 500. Flag the response shape.
5. **Collection-size caps (origin A.3).** My default vote: cap `events` per `/events` request (suggest ~500), and `transcripts`/`metadata` on `/complete` (suggest ~500 / ~50) → 400 on exceed, with a normalized code (suggest `request.too_large` — new, or reuse `cascade.invalid_audio`-style). Flag the limits + the code (I add it to ARCH-018 if new).
6. **`turn.not_found` code.** My default vote: a new `turn.not_found` (distinct from `session.not_found`) → 404, for a known session but unknown turnId. Flag if you'd rather reuse `session.not_found`.
7. **Extract `ResultExtensions` now?** My default vote: **yes** — `/complete`'s best-effort persist is the **second** `Result`-mapping consumer (the B.9c-i deferral condition), so extract a small `Result`/`Result<T>`→`IActionResult` helper (success→`Ok`/response, failure→sanitized `UiError`) shared by `/end` + `/complete`. Flag if you'd keep it inline.
