# /tdd brief — stale_session_flush

## Feature
Backend stale-session auto-end/flush (Flow H): when a new session is created while a prior session is still un-ended (the browser refreshed / abandoned it), the backend auto-ends + persists the abandoned session so it still produces its JSON evidence artifact.

## Use case + traceability
- **Task ID:** E.5-backend (the backend portion of E.5, pulled forward as the last backend slice before the frontend transition; the rest of E.5 — frontend WebRTC lifecycle — stays in E.5).
- **Architecture sections it implements:** `ARCH-017 §14 Flow H` (app refresh / recovery — "Backend stale-session auto-end/flush ensures abandoned sessions still produce a JSON artifact"), `ARCH-016` (persistence), `ARCH-010` (realtime lifecycle context).
- **Related context:**
  - Flow H (ARCH-017): "treat any in-progress turn as abandoned; … **Backend stale-session auto-end/flush ensures abandoned sessions still produce a JSON artifact**; completed turns are already safe via write-after-turn." So per-turn JSON is already written (C.4b/B.9c-ii); the gap is the **session-level finalize** (EndedAt + final summary + the session JSON) for a session the user never explicitly `/end`-ed.
  - **Reuse the existing end+persist path** — `SessionService.EndAsync` already finalizes (`SessionStore.End` → `EndedAt`) + persists via `SessionPersistenceWriter` + returns the persistence `Result` (degrade-not-crash, ARCH-016 / lesson §11). Do NOT duplicate the persist logic — route the flush through the same seam.
  - `SessionStore` (`Sessions/SessionStore.cs`): `Create` registers a new entry; `End(sessionId)` swaps `EndedAt`; `Get` reads. The store is pure in-memory (single-user, ConcurrentDictionary under a per-entry gate); persistence lives in `SessionService`.
  - **Single-trusted-user posture (ARCH-019):** at most one prior active session realistically; this is LOW-priority hardening (orphaned in-memory sessions are bounded by process lifetime), but the JSON-artifact guarantee is the point.

## Acceptance criteria (what "done" means)
- [ ] Creating a new session while a prior session is **un-ended** (`EndedAt == null`) **auto-ends + persists** the prior session (its final JSON artifact is written) before/as the new session is registered.
- [ ] The auto-end reuses the existing `EndAsync` finalize+persist seam (no duplicated persistence logic); the flushed session ends up `EndedAt`-stamped + summarized exactly as an explicit `/end` would.
- [ ] An already-ended prior session is **not** re-flushed (idempotent — only un-ended sessions flush).
- [ ] The **new session is always created successfully** even if a stale flush's persist fails — the flush degrades (persistence `Result` failure is swallowed/logged per lesson §11, never thrown; never blocks the new session).
- [ ] Completed turns are untouched (already write-after-turn safe); the flush only adds the session-level finalize.
- [ ] All new tests pass; backend suite stays green (311 → 311 + new).
- [ ] `/preflight` clean.

## RED test outline (Step 2)
1. **`create_flushes_prior_unended_session`** — create session A (un-ended) → create session B → A is now `EndedAt`-stamped **and** persisted (the writer was invoked for A); B is active. Why: Flow H artifact guarantee.
2. **`create_does_not_reflush_ended_session`** — A explicitly `/end`-ed → create B → A is not re-ended/re-persisted (idempotent; no second write). Why: only un-ended sessions flush.
3. **`flush_persist_failure_does_not_block_new_session`** — A un-ended, the writer fails for A → create B still succeeds (B returned, no throw); A's flush degrades (warning/log, lesson §11). Why: degrade-not-crash; the new session is never collateral.
4. **`create_with_no_prior_session_is_unaffected`** — first-ever create → no flush attempted, normal create. Why: no false-positive flush.
5. **(If Step-2.5 picks multi)** `create_flushes_all_prior_unended` — two un-ended priors → both flushed. Why: scope decision.

## Things to flag at Step 2.5
1. **Flush trigger.** On new-session `Create` (auto-end prior un-ended) vs a TTL/timer sweep vs an explicit endpoint. My default vote: **on `Create`** — deterministic, testable, single-user-appropriate; no background timer (overkill + harder to test). A TTL sweep is deferred G hardening.
2. **Scope — one prior or all un-ended?** Single-user → realistically one, but defensively flush **all** un-ended priors. Vote: **all un-ended** (cheap; robust to any leak).
3. **Persist-failure handling.** The flush's persist failure must not block the new session. Vote: **swallow + log/warn** (lesson §11 degrade) — the new `Create` always succeeds. (The flushed session's in-memory `EndedAt` still flips even if the disk write fails — partial-but-safe, mirrors `EndAsync`.)
4. **Order — flush before or after registering the new session?** Vote: **flush first** (so "active sessions" is clean when B registers), but either is fine since ids are unique; pick whichever reads cleaner under the gate.
5. **Concurrency.** The enumerate-then-end must be gate-safe (no torn read of the dictionary). Vote: enumerate ids under the store's existing locking discipline, then end each via the existing `End` path (its own gate), mirroring how `End` already swaps atomically.
