# /tdd brief — pricing_config_and_loader

## Feature
Populate `config/pricing.json` with the ARCH-014 starting block, extend `PricingOptions` from the A.2 minimal stub to the full pricing shape, and add a **file loader** (`PRICING_CONFIG_PATH` → deserialize via the shared `JsonDefaults`) that **degrades gracefully to "estimate unavailable" — never crashes — when the config is missing or invalid** (ARCH-018). This is the pricing data + load contract that B.5's `CostEstimator` consumes.

## Use case + traceability
- **Task ID:** A.4
- **Architecture sections it implements:** `ARCH-014` (Cost Estimation Model — the `pricing.json` structure + cost-estimate output), `ARCH-018` (missing/degraded config → graceful "estimate unavailable", never a crash).
- **Related context:** Depends on A.2 (`PricingOptions` minimal stub) + A.3 (shared `Common/JsonDefaults`, `Result`). **B.5** (`CostEstimator`) consumes this — it branches on pricing basis + computes per-turn/per-minute cost; A.4 only provides the data + the load/degrade contract. The DI wiring of `PRICING_CONFIG_PATH` is **A.5**.

## ⚠️ Key design wrinkle (read first)
`pricing.json` is a **standalone file** loaded from `PRICING_CONFIG_PATH` (ARCH-028: `PRICING_CONFIG_PATH=../../config/pricing.json`) — it is **NOT** an `appsettings` section. So `PricingOptions` is **deserialized from the file** (via `JsonDefaults.Options`), not section-bound via `IOptions`/`ConfigurationBinder` like the A.2 provider Options. **The A.2 `PricingOptions.SectionName = "Pricing"` is vestigial** — remove it (or repurpose only if A.5 genuinely needs a section path). This is the opposite load path from A.2; don't force it through `Configure<T>(GetSection(...))`.

## Things to flag at Step 2.5
1. **Load mechanism.** My default vote: **manual file-read + `JsonSerializer.Deserialize<PricingOptions>(json, JsonDefaults.Options)`** in `PricingLoader` — directly testable, reuses the A.3 shared options (lesson §2), and fits a standalone file (not an appsettings section). Not `AddJsonFile`/`Configure<T>`.
2. **Degradation representation.** My default vote: **`Result<PricingOptions>`** (A.3's `Result`) — `Success(options)` / `Failure("pricing config missing|invalid: …")`; B.5 maps `Failure` → "estimate unavailable". Clean, reuses A.3, no exceptions. (Alt: a flag on `PricingOptions` — but `Result` is the idiomatic A.3 choice.)
3. **Nested shape — dictionaries vs explicit per-model props.** My default vote: **dictionaries keyed by model name** (`Dictionary<string, RealtimeModelRates>` etc.) — a new model is a config-only change (matches the swappable-provider ethos); rate records carry the relevant nullable fields per basis. (Explicit per-model props would hard-code the model set.)
4. **`PricingLoader.cs` as a new file** (beyond A.4's "extended" list). My default vote: **yes, new `Cost/PricingLoader.cs`** — keeps load/degrade logic out of the POCO and reusable by B.5's `CostEstimator`. Confirm or fold into a static `PricingOptions.Load`.
5. **`gpt-5.4-mini` zero values.** My default vote: **keep ARCH-014 verbatim** (`0.0` + "CONFIRM at build" note) — do NOT invent values; the build-time confirmation is tracked (Decisions-tabled #4). Test #4 pins the placeholder is intact.
