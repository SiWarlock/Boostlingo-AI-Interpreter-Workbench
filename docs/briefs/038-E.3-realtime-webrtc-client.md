# /tdd brief — realtime_webrtc_client

## Feature
Stand up the browser Realtime transport **library**: mint the ephemeral client secret from our backend, a pure GA-event normalizer, and the WebRTC handshake shell (RTCPeerConnection + `oai-events` data channel + SDP exchange to OpenAI). E.3 builds the transport; E.4/E.5 wire it into turn control + lifecycle.

## Use case + traceability
- **Task ID:** E.3
- **Architecture sections it implements:** `ARCH-010` (§7 — WebRTC handshake, GA event mapping), `ARCH-007` (§4 — clean separation: components render only from the store; transports own all wire detail), `ARCH-009` (§6 — the `POST /api/realtime/client-secret` mint contract).
- **Related context:**
  - E.1 shipped the backend mint endpoint (`348ea76`): `POST /api/realtime/client-secret` `{sessionId, direction, model?}` → `{clientSecret:"ek_…", expiresAt(ISO-8601), model}`. Appendix A line ~1372 documents this as the `ClientSecret` DTO (`RealtimeTokenRequest`/`RealtimeTokenResponse`).
  - The shipped `http` boundary (`web/src/api/http.ts`, `request<T>`) + the api-client pattern (`sessionsApi`/`cascadeApi`) — web lessons §3 (base-URL-agnostic clients, single `http` failure boundary) + §4 (Vitest fetch-mock mechanics).
  - The cascade transport precedent: `cascadeStreamClient.ts`'s **pure** `dispatchCascadeMessage` router (web lesson §9) — E.3's `realtimeEvents` normalizer is the realtime analogue (pure classifier; the stateful store-dispatch is E.4, mirroring how the cascade store actions [§10] sat apart from the pure router).
  - D.4a (cascade stream client) is the **foundation-slice precedent**: a transport library landed first, wired into the UI by D.4b. E.3 follows the same shape (wired by E.4/E.5).

## SAFETY note (invariant #2 — read before Step 2.5)
The ephemeral `ek_…` is introduced to the **frontend** for the first time in this slice. It is used **transiently** to authorize the WebRTC SDP exchange (`Authorization: Bearer ek_…` to OpenAI) and must:
- **never** be written to `sessionStore` / `UiSessionState` / any persisted shape (root CLAUDE.md Key safety rule #2; ARCH-016). E.3 keeps it a **local** in the WebRTC client; it is NOT returned to the store layer and NOT logged.
- The standard provider key (invariant #1) **never** appears in `web/` — the frontend only ever holds the `ek_…` minted by our backend.

There is **no new persistence sentinel to add frontend-side** at E.3 (the frontend has no persistence path; the backend sentinel at E.1 already pins #2 on the wire). When E.4/E.5 wire realtime into the store, that slice adds a store-shape assertion that the `ek_` is absent. For E.3 the obligation is **honored by construction** + confirmed by the security-reviewer pass (see Estimated commit count).

## Acceptance criteria (what "done" means)
- [ ] `realtimeApi.mintClientSecret({sessionId, direction, model?})` → `POST /api/realtime/client-secret` (JSON), resolves the `{clientSecret, expiresAt, model}` response; a failure surfaces as `ApiError(UiError)` via the shared `http` boundary (no raw-body leak).
- [ ] `model` is **omitted** from the request body when undefined (backend resolves the default) — not sent as `model: undefined`.
- [ ] `realtimeEvents`: a **pure, stateless** normalizer — `parseRealtimeEvent(rawText)` guards malformed JSON (→ null, never throws); `normalizeRealtimeEvent(event)` classifies each GA event into a discriminated `NormalizedRealtimeEvent` (audioDelta / targetTranscriptDelta / sourceTranscriptDelta / sourceTranscriptCompleted / responseCreated / responseDone / error), accepts the legacy `response.audio.delta` alias, and returns `null` for unknown/malformed events.

- [ ] All unit tests in `web/src/api/realtimeApi.test.ts`, `web/src/realtime/realtimeEvents.test.ts`, `web/src/realtime/realtimeWebRtcClient.test.ts` pass.

## RED test outline (Step 2)

### `web/src/api/realtimeApi.test.ts`
1. **`mint_posts_to_client_secret_endpoint_with_json_body`** — Asserts: `POST /api/realtime/client-secret`, `Content-Type: application/json`, body = `JSON.stringify({sessionId, direction, model})`. Why: ARCH-009 §6 mint contract; web lesson §4 (assert request args directly — `expect.anything()` won't match an absent init).
2. **`mint_returns_parsed_token_response`** — Asserts: resolves `{clientSecret:"ek_…", expiresAt, model}` from a 200 JSON body. Why: ARCH-010 token shape; the `ek_` surfaces to the caller only.
3. **`mint_omits_model_when_undefined`** — Asserts: the sent body has no `model` key when `req.model` is undefined. Why: optional model (ARCH-009 §6); backend resolves the default.
4. **`mint_failure_surfaces_as_ApiError`** — Asserts: a non-OK status rejects with `ApiError` carrying a sanitized `UiError` (no raw body). Why: the `http` boundary (web lesson §3) — the mint path inherits no-leak.

### `web/src/realtime/realtimeEvents.test.ts`
1. **`parse_guards_malformed_json`** — Asserts: `parseRealtimeEvent('{not json')` → `null` (never throws). Why: cascade-dispatch guard precedent (web lesson §9 — guard the body, not just `JSON.parse`).
2. **`normalize_output_audio_delta`** — Asserts: `{type:'response.output_audio.delta', delta:'<b64>'}` → `{kind:'audioDelta', base64:'<b64>'}`. Why: ARCH-010 GA mapping (first one → `realtime.first_audio_delta`, computed in E.4).
3. **`normalize_legacy_audio_delta_alias`** — Asserts: `{type:'response.audio.delta', delta}` → `{kind:'audioDelta', …}`. Why: ARCH-010 explicit legacy-alias acceptance (dual-shape discipline, E.1 precedent).
4. **`normalize_output_audio_transcript_delta`** — Asserts: `{type:'response.output_audio_transcript.delta', delta:'hola'}` → `{kind:'targetTranscriptDelta', text:'hola'}`. Why: ARCH-010 (target transcript).
5. **`normalize_input_transcription_delta_and_completed`** — Asserts: `conversation.item.input_audio_transcription.delta` → `{kind:'sourceTranscriptDelta', text}`; `.completed` → `{kind:'sourceTranscriptCompleted', text}`. Why: ARCH-010 (source transcript; requires input transcription enabled — "source unavailable" handling is E.4's render concern).
6. **`normalize_response_lifecycle`** — Asserts: `response.created` → `{kind:'responseCreated'}`; `response.done` → `{kind:'responseDone'}`. Why: ARCH-010 response lifecycle (E.4 stamps `turn.completed`).
7. **`normalize_error_event`** — Asserts: an `error` / `response.error` event → `{kind:'error', …}` carrying a safe code (full `ProviderError` mapping is E.5; E.3 only classifies). Why: ARCH-018 (errors surfaced, never swallowed).
8. **`normalize_unknown_event_returns_null`** — Asserts: an unrecognized `type` → `null` (ignored). Why: forward-compat + guard-the-body (web lesson §9).

### `web/src/realtime/realtimeWebRtcClient.test.ts`
1. **`calls_url_is_ga_endpoint_without_model_query`** — Asserts: the SDP-exchange URL === `https://api.openai.com/v1/realtime/calls` with **no `?model=`**. Why: ARCH-010 §7 — appending `?model=` returns HTTP 400 (the highest-value pin; the bug only shows at live smoke otherwise).
2. **`exchange_sdp_posts_offer_with_ephemeral_bearer_and_sdp_content_type`** — Asserts: `exchangeSdpOffer(offerSdp, ek)` fetches the calls URL with `method:'POST'`, `Authorization: 'Bearer '+ek`, `Content-Type: 'application/sdp'`, `body === offerSdp` (raw string). Why: ARCH-010 §7 step 5; invariant #2 (the `ek_` used transiently to authorize — present in the header, nowhere persisted).
3. **`exchange_sdp_returns_answer_text`** — Asserts: resolves to `response.text()` (the answer SDP), NOT JSON-parsed. Why: the SDP exchange is `text/plain`, not the JSON `request` boundary.
4. **`exchange_sdp_non_ok_status_throws_typed_failure`** — Asserts: a non-2xx response → a typed sanitized failure (UiError-shaped, e.g. code `realtime.connect`), never a raw body. Why: ARCH-018 no-leak — the handshake failure surfaces sanitized.
5. **`exchange_sdp_fetch_rejection_throws_typed_failure`** — Asserts: a fetch rejection (network down) → typed failure, never a raw `TypeError`. Why: mirror the `http` fetch-rejection arm (web lesson §3).

## Things to flag at Step 2.5
1. **Where does the SDP exchange (`exchangeSdpOffer`) live — `realtimeWebRtcClient.ts` or `realtimeApi.ts`?** It POSTs to **OpenAI** (Bearer `ek_`, raw SDP, text answer), not our backend. `realtimeApi` is our-backend-JSON-only (the `request` boundary). My default vote: **`realtimeWebRtcClient.ts`** — keep `realtimeApi` for our backend only; the SDP exchange is a WebRTC-handshake-specific raw-text fetch.
2. **Does `normalizeRealtimeEvent` return a realtime-specific discriminated union, or directly emit `LatencyEvent`/`TranscriptSegment`?** `LatencyEvent`/`TranscriptSegment` need turn-relative timestamps/`relativeMs`/`clockSource` (stateful). My default vote: **a pure `NormalizedRealtimeEvent` union** (semantic kind + payload only) — E.4's turn-control layer owns the stateful first-event stamping + `LatencyEvent` construction (browser clock) + store dispatch. Mirrors the cascade split (pure `dispatchCascadeMessage` router §9 vs. stateful store actions §10).
3. **How much of the RTCPeerConnection orchestration to test vs. leave manual-smoke?** My default vote: **leave the `createRealtimeWebRtcClient` factory a manual-smoke shell**; pin only the pure seams (`realtimeCallsUrl`, `exchangeSdpOffer` via fetch-mock). Root CLAUDE.md lists WebRTC as manual-smoke-exempt (unlike WebSocket, which cascade fake-tested at §9) — RTCPeerConnection's SDP/ICE/track/DC surface is genuinely browser-internal; a fake re-implements the browser without pinning real behavior. (If you spot a cheap, high-value fake-`RTCPeerConnection` teardown-order test, add it — not required.)
4. **Capture surface for the realtime path** (carry-forward KEEP, origin D.3). Realtime wants the raw mic `MediaStreamTrack` (`pc.addTrack`), NOT the cascade `audioCaptureController.onFrame` linear16 frames. My default vote: **E.3's WebRTC client calls `getUserMedia({audio:true})` itself** and `addTrack`s the raw track — minimal, clean-separation-preserving (the controller's worklet/linear16 machinery is irrelevant to WebRTC). Revisit a shared `getUserMedia` helper only if duplication bites. `getUserMedia` itself is browser-internal (manual-smoke); only pure helpers around it are tested.
5. **Legacy alias scope.** ARCH-010 explicitly aliases `response.audio.delta` ≡ `response.output_audio.delta`. My default vote: **handle the audio alias** (pinned by test 3). If the live GA also uses a legacy transcript alias (`response.audio_transcript.delta`), alias it symmetrically — but confirm the exact `type` strings at first smoke (flag as an architecture-doc note, don't guess beyond ARCH-010's explicit list).
