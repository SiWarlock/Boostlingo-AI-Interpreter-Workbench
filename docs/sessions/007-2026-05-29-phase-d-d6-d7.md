### The load-bearing design — D.6 three-source metrics model

Three metrics sources, never conflated:
1. **Per-stage cascade latency** — from the store's `currentTurn.latency.stages` map (server-computed `relativeMs`), passed through, **never recomputed** (web LESSONS §7).
2. **Top-level client-timing deltas** (`speechEndToFirstAudioMs`/`speechEndToPlaybackMs`/`totalTurnMs`) — **frontend-computed** via absolute-timestamp subtraction, because for cascade the backend structurally *cannot* (no client→server latency channel). `deriveTurnMetrics` mirrors the backend `MetricsAggregator.Between` (`to.timestamp − from.timestamp`, `undefined` if either endpoint absent, **never clamped** — cross-clock skew is disclosed per ARCH-013, never hidden; **never uses `relativeMs`** for cross-event math). The frontend's browser-clock markers — `turn.recording.started/stopped`, `playback.started` (D.5), and the new `turn.completed` — are the only source. `turn.completed` had to be stamped client-side (in `completeTurn`, on the WS `done`) because the backend stamps it on finalize but does **not** stream it over the cascade WS; it is the canonical `totalTurn` terminal, with `tts.complete` (server-clock) as the fallback.
3. **Session averages by mode** — from the backend `GET /api/sessions/{id}/summary` (canonical), fetched on turn-completion + manual refresh, held on `state.summary`.

## Decisions made

- **Store retention of raw latency events + full cost (D.6).** `deriveTurnMetrics`'s absolute-timestamp math + the CostPanel's assumptions tooltip both need data the store was discarding (`appendLatencyEvent` kept only `relativeMs`; `setTurnCost` kept 3 scalars). Added `TurnViewModel.latencyEvents[]` + `cost` (additive/optional → zero breakage). Required for reachability — without it the feature is tested-but-unwireable. Orchestrator-approved at Step 2.5.
- **`turn.completed` stamped inside `completeTurn` (D.6).** Centralized (no caller can forget it), co-located with `completedAt`, lands before the turn moves to `turns[]`. `totalTurn` is then browser-clean (`recording.started → turn.completed`) with `tts.complete` (server) as the documented fallback.
- **`getSummary` fetch in an App-shell effect (D.6).** Mirrors the existing config-bootstrap effect (App owns the fetch; panels stay dumb store projections); on turn-completion + a manual Refresh. Held on `state.summary` so the MetricsPanel renders only from the store (ARCH-007).
- **`errorCopy` reads only `code` (D.7).** Returns fixed copy per code (never echoes `safeMessage`) — the structural never-raw guarantee (ARCH-007/018), even though `safeMessage` is already sanitized. Pinned by the `RAW-PROVIDER-LEAK` sentinel test.
- **Per-file `// @vitest-environment jsdom` (D.7).** The 2 component tests opt into jsdom per-file; the node default is preserved so the 92 existing tests (fetch/FormData/Blob as node globals) are untouched. `setupFiles` only registers jest-dom matchers (DOM-free at import); RTL `cleanup` is per-jsdom-file.
- **Mic-denied asserts the truthful settled state (D.7).** The brief/MVP_TASKS said "Start disabled"; tracing the D.3/D.4b path showed mic-denied settles to `turnStatus='failed'`, which `canStartRecording` *allows* (retry) — and the actionable copy literally says "…and retry." So the test asserts: error surfaced + Stop disabled (not stuck mid-record) + **Start re-enabled for retry**. Lead-approved; ARCH-020 + MVP_TASKS D.7 wording clarified by the orchestrator.

## Decisions explicitly NOT made (deferred)

- **`micDenied` gate on Start (D.7).** Explicitly NOT added — it would contradict the "…and retry" copy and is out of the render-test scope. Mic-denied stays a retryable failed turn.
- **`completeTurn` failed→completed status overwrite (D.6).** A `done{completed}` following an `error` frame would overwrite `'failed'` — but the C.4b backend contract sends `done{failed}` on failure (and `error` is terminal), and an existing test pins the current behavior. Deferred → G hardening (backend-contract-protected).
- **Unbounded store growth (D.6).** `latencyEvents[]` per-turn + `turns[]` per-session have no cap. Single-trusted-user (ARCH-019); bounded by the turn/session lifecycle. Deferred → G.4 (5-min no-leak), mirroring the backend's bounded-channel deferral.
- **Per-error dismiss on the ErrorBanner (D.7).** Render-only for D.7; `clearErrors` on new-session already exists.
