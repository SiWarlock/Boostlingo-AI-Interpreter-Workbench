## Decisions made

- **External-store pattern** (D.1): a pure `createSessionStore()` factory (unit-tested) + a thin `useSyncExternalStore` hook (manual-smoke); immutable state (new ref per action) so the hook re-renders and never loops. The store is the single error sink.
- **`http` as the single failure boundary** (D.1, hardened D.1 reviewer): non-OK status, fetch-rejection, AND an unparseable 2xx body all → `ApiError(UiError)`; never leaks a raw body/`TypeError`/stack.
- **Pragmatic-accrete types** (D.1): mirror only what each slice consumes; the 4 streaming wire types graduated out of the opaque `InterpretationTurn` at D.4a when the WS dispatch needed them.
- **Config-gating as pure selectors** (D.2): components stay thin (render the selector result + dispatch intents); the store is the live config source-of-truth via one merging `updateSessionConfig` — **guarded** so it transitions `idle→configured` only and never drags a started session backwards (the D.2 Flow-G HIGH fix).
- **DI'd orchestration modules** (D.2 `sessionActions`, D.4b `recordingActions`): tested against the real store + mocked api/client/capture; both error arms + ordering + guards pinned without a render.
- **Capture = manual-smoke shell over pure helpers** (D.3): linear16 conversion / mimeType probe / mic-error map are test-first; getUserMedia/AudioWorklet/MediaRecorder are manual-smoke. The worklet MUST `connect(context.destination)` or `process()` never fires (the D.3 HIGH fix).
- **WS client TDD'd against an injected fake `WebSocket`** (D.4a): full lifecycle (open→start→frames→stop→dispatch→done/close) unit-tested; raw audio routes to `onAudio`, NEVER the store (invariant #3). Lifecycle hardening: a server `error` frame is terminal; `teardown()` on re-start; guard the dispatch switch body; the D.4b pre-open frame-queue + idempotent deferred stop; the `inFlight` concurrent-start guard.
- **Streaming transcript normalization in the store** (D.4a): replace trailing non-final partial / finalize on `isFinal` / new entry after a final — so the panel renders without its own partial-tracking.
- **`appendLatencyEvent` stages-only** (D.4a): per-stage timing keyed by event name; the top-level speech-end deltas are NOT computed client-side (the backend MetricsAggregator is canonical, lesson §7).
- **`ttsVoice: ''` (blank)** (D.4b): lets the backend `ResolveVoice` own per-target-language voice resolution — no `DEFAULT_TTS_VOICE` constant, no store field (there's no UI voice selector).
- **Playback = manual-smoke MSE/HTMLAudioElement shell over pure helpers** (D.5): MSE-first progressive + best-effort blob fallback; `playback.started` stamped on the `playing` event (once/turn); `clampAudioContentType` defense-in-depth at the data boundary; `onAudio` settable-delegate closes the D.4b deferral without reconstructing the singleton.

## Decisions explicitly NOT made (deferred)

- **Top-level speech-end metrics computation** → D.6: must be a delta of absolute browser timestamps (NOT the `relativeMs:0` placeholder — streaming-honesty), and needs a frontend `turn.recording.stopped` browser-clock marker (not stamped in D.4b).
- **Cascade `playback.started` persistence** → F.3: it's a frontend-display-only marker; the cascade WS has no client→server latency-report channel (that's realtime's `POST …/events`, E), so the persisted/aggregated cascade summary shows speechEnd→playback `n/a` for cascade. Accept + document (G.5); F.3 decides whether to add a channel.
- **Blob-fallback completeness** → G: an HTMLAudioElement source is static, so the live-blob fallback can't progressively play — a `done`-driven `flush()`/end-of-stream assemble would complete it. MSE/Chrome is the demo path; low-priority G hardening if non-MSE browsers matter.
- **Component render tests** → D.7: the two PRD-aligned component tests (mode-toggle-disabled + mic-denied) land in D.7 with jsdom + Testing Library. D.6's panels + D.7's ErrorBanner remain.
- **D.2 stale-errors on config-mutation:** decided to keep store actions orthogonal — errors clear via explicit `clearErrors()` (the D.2 Start flow calls it). No auto-clear coupling in `updateSessionConfig`.

## How to use what was built

Run the backend (`dotnet run --project server/AiInterpreter.Api`, port 5179) + `cd web && npm run dev` (Vite on 5173, `/api` proxied + `ws:true`). With provider keys configured, the cascade mode is enabled (config-gating); pick direction/translation-model, Start → speak → Stop. Source partials render live, target tokens + cost stream into the store, and TTS audio plays (MSE in Chrome). Panels (D.6) will surface the metrics the streaming actions already populate.
